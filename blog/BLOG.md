## 1. 정체성 — 재무제표의 "왜?" 를 풀어내는 엔진으로 간다

dartlab 블로그는 재무제표의 **"왜?"** 를 풀어내는 엔진이다.

재무제표를 열었더니 이상한 게 보이고, 파고 들어가면 **"아 그래서"** 하는 발견의 쾌감. "이 회사 부채 37 조인데 빚이 0 원" · "매출 1.6 조인데 점유율 52%" · "영업이익률 4%→18%, 공장 안 지었는데 왜?" — 이게 블로그의 정체성이다.

**핵심 약속**:
- 공시 원문을 직접 읽고 판단하는 **실전 교재**
- dartlab 없이도 가치 있는 글이어야 함
- 숫자는 **정직성 최우선**
- 독자가 "나도 재무제표 열어보고 싶다" 고 느끼면 성공

## 3. 제작 파이프라인 — 5 에이전트 체인으로 돌린다

모든 기업이야기는 단일 파이프라인으로 간다.

### Phase 0: 데이터 탐색

dartlab **5 엔진을 직접 돌려** "흥미로운 것, 의외인 것, 모순" 을 찾는다. **사람이 아는 이야기가 아니라 데이터가 말하는 이야기**를 먼저 찾는 게 핵심.

```python
# 1) scan — 동종사 상대 위치
dartlab.scan("profitability"); dartlab.scan("growth"); dartlab.scan("quality")
dartlab.scan("capital"); dartlab.scan("debt"); c.governance()

# 2) analysis — 14축 분해
c.analysis("수익구조"); c.analysis("비용구조"); c.analysis("이익품질")
c.analysis("자본배분"); c.analysis("자금조달"); c.analysis("현금흐름")

# 3) quant — 시장 평가
c.quant("종합"); c.quant("팩터")
```

→ "이 회사에서 발견한 흥미로운 것 5~10 개" 리스트 → 관통선 후보 선택.

**SEO 경쟁 분석**: "회사명 + 분석" 네이버 상위 5 개 글의 H2 비교 → 내 글에 없는 관점 목록화.

**반복 실패** — 단독 작업으로 에이전트 체인 건너뛰면 기획·해석·데이터 중 한 축이 무너진다.

### Phase 1: 기획 (4 에이전트 병렬)

| 에이전트 | 역할 | 산출물 |
|---|---|---|
| **기획자** | 핵심 질문 1 개, 관통선, 막 구조 | 막 구조표 |
| **재무분석가** | dartlab 실행, "이상한 숫자" 5 개 발굴 | 막별 테이블 |
| **스토리작가** | 구체적 장면, 인물 대사, 리듬 | 장면 5 개 |
| **마케터** | 제목 5 개, description 3 개 | 최종 선택지 |

### 관통선 — 하나의 인과로 1 막부터 끝까지 꿰뚫는다

90 점 이상의 글은 관통선이 **하나의 인과** 로 모든 막을 꿰뚫는다.

- 관통선 = **"왜 A 인데 B 인가?"** 형태의 질문 1 개.
- 매 막의 끝이 다음 막의 원인.
- 관통선 무관 토픽은 별도 막 없이 **관련 막에 1~2 문단**으로 녹인다.
- 검증: 모든 막의 제목 나열 후 "이 막을 빼면 관통선이 끊기는가?" → 끊기지 않으면 삭제·흡수.

**사례**:
- ✓ SK하이닉스: "5 번 죽을 뻔한 회사가 왜 매번 더 강해졌는가"
- ✓ 대한조선: "조선소인데 OPM 24% 가 어떻게 가능한가"

**반복 실패** — 병렬 서사 ("세 겹의 재무제표" 같은) 는 85 점 천장. 현대글로비스 사례. 관통선 하나로 꿰야 90 점+.

### Phase 2: 집필 — 재무분석가 → 스토리작가 순차로 간다

- 매 막을 **재무분석가 (데이터 + 해석) → 스토리작가 (장면 + 리듬)** 순서.
- 매 막 "왜?" 로 시작, **관통선 질문의 답을 향해** 흐르기.
- **매 막 끝에 다음 막으로의 인과 다리 1 문장** 필수.

### 막 개수 — 이야기가 요구하는 만큼 쓴다

- **6 막은 기본 템플릿, 고정 아님.** 이야기거리가 많으면 7~8 막까지 확장 가능.
- 확장 기준: 막 하나가 3,500 자 넘어가면 두 막으로 쪼개기, 관통선이 흐려지면 합치기.
- "막 개수 맞추느라 이야기 잘라내기" 를 하지 않는다. 이야기가 요구하는 구조가 우선.

### 회사마다 구조가 다르게 간다 (막 뼈대 복붙 금지)

**"1 막 도입 → 2~5 막 본론 → 6 막 체크포인트" 뼈대를 회사마다 똑같이 복사하지 않는다.** 회사마다 특징이 다르면 구조도 달라야 한다.

| 회사 유형 예 | 구조 변형 |
|---|---|
| **팔란티어** | 주식보상비용(SBC) 혼자 $684M 이 이야기 전부 → SBC 딥다이브 막을 독립으로, 나머지 짧게 |
| **한미약품** | 창업자 임성기 40 년 + 2016.9.29 베링거 공시 타이밍 → 인물 막 1 개 + 다큐 장면 막 1 개 추가 |
| **효성화학** | 특수가스 매각의 회계 처리가 복잡 → 회계 딥다이브 막 독립 |
| **SK하이닉스** | 30 년 사이클 5 번 → 시간선이 뼈대 → 시대별 막 5~6 개 (6 막 틀 깨짐) |
| **쿠팡** | 현금흐름표가 전부 → 매출·이익 막 축소, CF 막 3 개로 확장 |

**"1 막→6 막" 공식 대신 관통선이 요구하는 구조로 조립한다.** 6 막은 시작점일 뿐, 최종 뼈대는 글마다 다르다.

### 풍부함 축 8 개 — 최소 3~4 개 적용한다

| # | 축 | 적용 예 |
|---|---|---|
| 1 | **막 수 확장** | 6 막 → 8~10 막. 인물·경쟁사·리스크·산업 지형 독립 막 분리 |
| 2 | **인물 서사 독립 막** | 창업자·CEO 1 명의 삶 × 재무제표 시계열 교차 |
| 3 | **역사 사건 막** | 공시 한 건이 회사의 운명을 바꾼 장면 재현 ("2016.9.29 오전 9 시 아침 공시") |
| 4 | **경쟁사 심층 막** | "셀트리온은 왜 26% 마진인가" 를 3~4 문단 해부 |
| 5 | **산업 지형 막** | 한국 NCC 통폐합 논의 · 중국 증설 · FDA 규제 |
| 6 | **외부 쇼크 흔적 막** | 코로나 · 금리 · 환율 · 전쟁 · 원자재 사이클이 재무제표에 남긴 발자국 |
| 7 | **기술·제품 원리 막** | 에소메졸 복제약 방어 / NCC 톤당 200 달러 스프레드 |
| 8 | **투자자 시각 막** | 대주주·배당·자사주·밸류에이션 독립 |

**축별 풍부함 기법**:
- **인물 인용 박스** — 사업보고서·인터뷰·주주총회 발언 실제 문구 인용 (> blockquote).
- **공시 원문 박스** — DART 전자공시 특정 문구를 그대로 인용 ("◆click◆ 증자(감자)현황" 같은 원본 리터럴).
- **시간 다큐 문체** — "2016 년 9 월 29 일 오전 9 시, 한미약품은…" 같은 시점 구체 재현.
- **H3 수 확장** — 1 막당 H3 3~5 개 → **H3 6~10 개**.
- **경쟁사 코드 실행** — 같은 dartlab 호출을 경쟁사에 적용하고 결과 대조.
- **용어 사전 섹션** — 글 말미에 "이 글에 나온 제약 용어 10 개" 정의.

**분량 상한**: 기본 20,000 자 → 풍부함 축 4 개 이상 적용 시 **25,000~30,000 자**.

### 자가 진단 체크리스트 (발행 전)

- [ ] 이 글의 막 구조가 이전 3 편과 동일한가? → Yes 면 최소 1 축 더 확장
- [ ] 인물 이름이 3 명 이상 등장하는가? → No 면 인물 서사 보강
- [ ] 공시 원문 또는 실제 발언이 1 건 이상 인용됐는가? → No 면 보강
- [ ] 경쟁사 비교가 "표 1 개 + 1~2 문단" 에 그치는가? → Yes 면 경쟁사 심층 막 검토
- [ ] 독자가 "다음 한미약품 글은 어떻게 생겼을지" 예측 가능한가? → Yes 면 구조 파괴 검토

**반복 실패** — #46 네이버~#57 한미 12 편이 전부 같은 6 막 형식 → 독자가 펼치자마자 전개 예측 가능 → 놀라움 소멸. 분량 채우려고 같은 말 반복하거나 불필요한 부연 금지.

### 과거~현재 패턴 + 산업 패턴 + 투자 포인트 막 (회사편 필수)

모든 기업이야기 회사편에 **1 막** 포함한다. 위치는 자연스러운 곳 (마지막 직전 또는 6 막 내부 섹션). 내용 3 가지:

1. **이 종목의 과거~현재 패턴** — 이번 사이클이 처음인지 반복인지, 과거 유사 국면에서 어떻게 돌아왔는지. 9 년 시계열에서 반복 패턴을 추출.
2. **산업 패턴** — 소속 산업의 사이클·경쟁 구조·변곡점. 한국 종목은 `c.industry` 엔진 활용 가능. US 종목은 10-K Risk Factors + 산업 보고서 인용.
3. **투자 포인트** — 이 종목을 보는 사람이 다음 분기·1 년에 봐야 할 체크포인트 3~5 개. "로보택시 분기 매출" · "CAPEX/매출 비율" · "유효세율" 같은 구체 지표.

⚠ 현재 `industry` 엔진은 한국 전용. EDGAR 종목 (META·TSLA 등) 은 외부 출처 + 질적 분석으로 대체.

### 시나리오 분석 — 관통선 강화 시에만 선별로 넣는다

시나리오가 관통선의 답을 **직접 강화** 하는 글에만 선별 허용.

**넣는 기준**: "이 변수가 바뀌면 마진·매출이 구조적으로 바뀐다" 가 관통선 자체일 때.
- ✓ 오뚜기: "해외 비중 30% 가 되면 OPM 5.7%" — 관통선(내수 천장 5%) 의 직접 답.
- ✓ 에스퓨얼셀: "분기 매출 100 억이면 원가율 100% 이하" — 생존 조건 자체.

**넣을 때 규칙**:
- 독립 H2 없이 **6 막(체크포인트) 안에 H3 으로 녹인다**.
- 본문 서술로 "이 변수가 X 가 되면 Y 가 Z 로 바뀐다" 한두 문단.
- 시나리오 숫자도 **검증표에 "시뮬레이션" 명시** (dartlab 실측과 구분).
- 글의 닫힘성 우선 — 시나리오가 "다음에 봐야 할 것" 으로 끝나지 않고 **판단으로 닫아야 함**.

**반복 실패** — 전편에 "시나리오 분석" H2 반복 → 템플릿 복사. 추정 숫자가 검증 없이 본문에 → 정직성 위반. "낙관·기본·비관" 3 열 표 → 리서치 보고서 톤. 한화오션·SK하이닉스 관통선과 무관한 시나리오 나열 피한다.

## 4. 품질 기준 — 5 약속으로 간다

1. **제목이 후킹이다** (60 자 이하, 검색형 질문 또는 판단 문장).
   - ✓ "SK하이닉스 (000660) — 한국 반도체 30 년의 미스터리, 그리고 영업이익률 58%"
   - ❌ "SK하이닉스 분석"
2. **첫 2 문단 안에 핵심 답** (200~300 자) → Featured Snippet + LLM 인용 직결.
   - 도입부 관성 금지. "이상한 재무제표" 프레임은 최대 1 회.
   - 매 글 도입부는 **그 기업만의 고유한 장면·숫자·질문**.
3. **단일 핵심 질문이 글 전체 지탱**.
4. **고유 H2 구조** — 템플릿 복사 금지.
5. **닫힘성** — 이 글에서 할 말을 이 글에서 끝낸다.

### 분량 가이드

| 글 종류 | 권장 분량 | 막 평균 | 한 막 자산 |
|---|---|---|---|
| 회사 종합편 | **20,000+ 자** | 3,000~4,000 자 | 표 + SVG + 이미지 |
| 주제 심화편 | 10,000~15,000 자 | 2,000~3,000 자 | 표 + SVG |
| 공시 읽기 가이드 | 5,000~9,000 자 | 800~1,200 자 | SVG 위주 |
| 소식·입문 | 3,000~6,000 자 | 500~800 자 | SVG 1~3 개 |

**목표: 각 막에 표 1 개, SVG 1 개, 이미지 1 개 자연스럽게 = 위 분량 자동 충족**.

### 20,000 자의 의미 — 글자수가 아니라 읽을거리다

**독자가 이 글을 읽고 나서 "이 회사에 대해 충분히 이해했다" 고 느끼는 데 필요한 분량**이 20,000 자.

15,000 자 이하의 글은 대부분 이 중 하나가 빠져있다:
- 경쟁사·업종 비교가 표면적 (숫자만 나열하고 "왜 다른가" 설명 없음)
- 과거 패턴이 1~2 줄로 끝남 (9 년 시계열을 가졌는데 3 년만 봄)
- 메커니즘 설명 없음 ("원가율이 올랐다" → 왜 올랐는지 안 씀)
- 인물·의사결정 서사 없음 (숫자만 있고 사람이 없음)
- 산업 맥락 빈약 (이 회사만 따로 떼어놓고 분석)

**20,000 자를 자연스럽게 채우는 방법**: 매 막에서 "왜?" 를 한 번 더 파고, 비교 대상을 구체적으로 대조하고, 독자가 "그래서 나는 뭘 봐야 하는데?" 에 답하면 된다.

**반복 실패** — 글자수 채우기 위해 같은 말 반복·불필요한 부연. 분량은 이야기가 요구하는 만큼.

| 4 | 숫자 | 검증된 dartlab 시계열 |

### 콘텐츠 톤 — 중립·진짜 위험·가격 표현 없음

| 주제 | 이렇게 한다 |
|---|---|
| 숫자 자랑 대신 효용 | "수익성·안정성·현금흐름 등 시장 전체를 읽는다" |
| 부채비율 중립 | ICR<1·유동비율 급락 같은 진짜 위험만 위험으로 |
| 가격 표현 0 건 | provider·API 가격 언급 자체를 하지 않는다 |
| 이 글에서 끝낸다 | "다음 글에서" 식 예고로 미래 글에 의존하지 않는다 |

**반복 실패** — "14 축 체계" · "19 개 분석" · "6 막 구조" 같은 내부 스펙 나열 → 광고처럼 보임. 부채비율을 무조건 위험으로 포장. "무료" 표현 · provider 가격 언급. "다음 글에서 다룬다" — 존재 안 하는 글 링크는 신뢰 깎음.

### 각 막 — 왜?로 시작해 판단으로 끝난다

- **매 막 "왜?" 로 시작**.
- **표 앞에 맥락 1 줄 필수** — 숫자 먼저 쓰지 않는다.
- **막 전환 시 인과 연결 1 문장** — "3 막의 결과가 4 막을 가능하게 한 것이다".
- **마지막 막은 판단으로 끝난다**.
- **각 막에 H3 소제목 3~5 개** — 글 전체 H3 합계 15 개 이상 목표.
  - 형식: `### 매출원가율 141.6% — 팔수록 적자` (숫자 + 한 줄 해석).
- **각 막에 dartlab 코드 + 표 1 개 이상**.

좋은 H3 예: LG이노텍 23 · 한온시스템 18 · HDC현산 26.

**반복 실패** — 중복 막 / 체크리스트·나열로 닫는 마지막 막 / H3 없이 본문만 이어져 AI 인용 단위 부재.

### 용어 풀어쓰기 — 같은 문장 안에서 푼다

독자는 재무제표를 처음 보는 사람. **"OPM" 은 본문에서 쓰지 않는다** — "영업이익률" 로 쓰고 첫 등장 시 "(매출 대비 영업이익 비율)" 괄호 풀이. 표 헤더에서만 "OPM (%)" 허용.

| 쓰지 않는다 | 이렇게 쓴다 |
|---|---|
| OPM 6.2% | 영업이익률 6.2% (매출 대비 영업이익 비율) |
| GP 마진 44% | 매출총이익률 44% (물건 팔고 원가 빼면 44% 남는다) |
| OCF | 영업활동현금흐름 (실제 장사해서 들어온 현금) |
| FCF | 잉여현금흐름 (영업현금에서 투자비 빼고 남는 돈) |
| CCC -55 일 | 현금전환주기 -55 일 (물건 팔기 전에 돈이 먼저 들어오는 구조) |
| ICR 0.92x | 이자보상배율 0.92 배 (번 돈으로 이자도 못 갚는 상태) |
| ROIC · ROE | 투하자본수익률 · 자기자본수익률 |
| 운전자본 | 영업에 묶여있는 돈 (재고+매출채권-매입채무) |
| 감가상각 | 과거에 산 설비 값을 매년 조금씩 비용으로 깎는 것 |

**규칙**: 괄호 풀이 1 줄 필수, 첫 등장 시 풀이 후 이후 약어 OK, 비유 권장.

**반복 실패** — 독자가 "OPM 이 뭔데?" 라고 묻는 순간 이 글은 실패한 것. GPM·ROE·ROA 도 첫 등장 시 반드시 한국어+괄호 풀이.

## 5. dartlab 호출 — 8 패턴으로 간다

블로그에 자주 등장하는 호출. 본문 코드블록에 그대로 박는다 — **독자가 복사 → 실행 → 검증 가능**.

### 1) IS 9 년 시계열 (1 년 합산)

```python
import dartlab
c = dartlab.Company("000660")
c.select("IS", ["매출액","매출원가","매출총이익","판매비와관리비","영업이익","당기순이익"])
# 분기 컬럼 반환 → 1년치 합산 필요
df = c.select("IS", ["매출액"])
row = df.row(0, named=True)
annual = {y: sum(row.get(f"{y}Q{q}", 0) or 0 for q in range(1, 5)) for y in range(2017, 2026)}
```

⚠ 분기 단독값과 1 년 합산을 섞지 않는다.

### 2) BS Q4 스냅샷

```python
c.select("BS", ["자산총계","부채총계","자본총계","현금및현금성자산"])
# BS는 시점 데이터 → 합산 X, 각 연도 Q4만
```

### 3) CF 9 년 시계열 + 잉여현금 파생행

```python
c.select("CF", ["영업활동현금흐름","유형자산의 취득","배당금지급"])
# CF − CAPEX = 잉여현금 — 본문 표에 반드시 포함
```

### 4) 비율 분기 시계열

```python
c.select("ratios", ["영업이익률 (%)","매출총이익률 (%)","부채비율 (%)"])
```

⚠ `c.analysis("financial","수익성")["marginWaterfall"]` 는 1 년치. 섞지 않는다.

### 5) 수익성 분석

```python
prof = c.analysis("financial", "수익성")
prof["marginWaterfall"]["history"][0]      # 1년치 마진 분해
prof["roicTree"]["history"][0]             # ROIC 트리
prof["penmanDecomposition"]["history"]     # Penman RNOA/FLEV/NBC/Spread
```

`marginDriver` 는 dartlab 자체 분류이므로 코드블록 명시 인용 가능:

```
"marginDriver": "높은 가격결정력 (매출총이익률 > 40%)"
```

### 6) 자금조달

```python
fund = c.analysis("financial", "자금조달")
fund["capitalOverview"]   # 표시용 요약 (총자산/총부채/자기자본/순차입금만)
fund["fundingSources"]["notesDetail"]["borrowings"]  # 차입금 항목별
```

⚠ `capitalOverview` 의 "순차입금" 은 내부 정의. "총 차입금" 필요하면 `notesDetail.borrowings` 의 "합계" 행 사용.

### 7) 자본배분 — 배당 (두 출처 차이)

```python
ca = c.analysis("financial", "자본배분")
ca["dividendPolicy"]["history"]  # CF dividendsPaid 1년치 합산

c.report.dividend.dps  # 주당 현금배당금 (DART 정기보고서)
```

⚠ 두 출처는 다를 수 있음 (CF = 자본 재구성·우선주 포함, report = 보통주만). 어느 출처를 썼는지 검증표에 명시.

### 8) 신용등급

```python
cr = c.credit("등급")
cr["grade"]; cr["score"]; cr["healthScore"]; cr["divergenceExplanation"]
```

### 분기 vs 연간 vs Q4 — 표 헤더 단일 라벨

| 데이터 종류 | 단위 | 표 헤더 라벨 |
|---|---|---|
| IS (매출·원가·이익) | 1 년치 합산 | `(1년치 합산, 조원)` |
| BS (자산·부채·자본) | 시점 스냅샷 | `(Q4 스냅샷, 조원)` |
| CF (현금흐름) | 1 년치 합산 | `(1년치 합산, 조원)` |
| 비율 | 분기 | `(분기, %)` |
| marginWaterfall | 1 년치 | `(1년치 분해, %)` |

**한 표 안에 여러 단위를 섞지 않는다.**

### 회사 종합편 필수 6 데이터

| # | 데이터 | 호출 | 본문 위치 |
|---|---|---|---|
| 1 | IS 9 년 시계열 표 | `c.select("IS", [...])` 분기 합산 | 수익성·마진 막 시작 |
| 2 | BS 9 년 Q4 스냅샷 | `c.select("BS", [...])` Q4 만 | 안정성·자금조달 막 시작 |
| 3 | CF 9 년 + 잉여현금 파생행 | `c.select("CF", [...])` 분기 합산 | 현금·자본배분 막 시작 |
| 4 | 핵심 비율 분기 시계열 1~3 개 | `c.select("ratios", [...])` | 핵심 변곡점 막 |
| 5 | dartlab 자체 분석 인용 ≥1 | marginDriver / flag / divergence | 메커니즘 막 |
| 6 | 신용등급 카드 | `c.credit("등급")` | 안정성 막 |

### 표 작성 규칙

1. **컬럼 = 시점, 좌측이 가장 최신** (2025 → 2024 → … → 2017).
2. **첫 컬럼 = 항목명**, 짧고 명확하게.
3. **단위는 표 헤더에 명시**.
4. **숫자 컬럼 우측 정렬** (`---:`).
5. **강조 `**굵게**`** — 정점·바닥·사상 최대.
6. **표 아래 한 줄 풀이** — `표시: **+58.40** = 사상 최대`.

### 단위 결정

모든 행이 0.5~1000 범위에 들어가는 단위.

| 회사 | 단위 |
|---|---|
| 대형 (SK하이닉스 · 삼성전자) | 조원 |
| 중형 (삼양식품) | 억원 |
| 비율 | % |

같은 글에서 단위 혼용 OK (표마다 헤더에 명시).

### 수치 정직성 — 6 원칙으로 간다

지난 라운드 19 건 거짓의 80% 가 이 6 가지 중 하나를 어겼다.

1. **분기 vs 연간 라벨을 섞지 않는다**
   - ❌ `| 연도 (Q4) | 매출 |` ← Q4 단독값인데 "2025" 로 라벨
   - ✓ `| 항목 (Q4 스냅샷, 조원) | 2025Q4 | 2024Q4 |`
2. **모든 인용 수치는 검증표에 등록한다**
   ```markdown
   | 본문 수치 | dartlab 호출 | 결과 |
   |---|---|---|
   | 2025 연간 매출 97.15조 | `c.select("IS",["매출액"])` 분기 합산 | ✓ 실측 |
   ```
   검증표에 없는 숫자가 본문에 있으면 발행 차단.
3. **핵심 후킹 수치는 두 번 검증한다** (작성 시 + 발행 직전 dartlab 재실행).
4. **dartlab 자체 출력은 직접 인용 가능하지만 인용임을 명시한다** (코드블록 또는 따옴표).
5. **"약 X 조" · "거의 Y 배" 어림 표현은 정확값 옆에만 쓴다**
   - ✓ "27.52 조 (2018 정점의 약 1.7 배)"
   - ❌ "약 30 조 정도가 1 차 정점의 두 배에 가깝다"
6. **거짓 발견 시 5 곳 같이 갱신한다** (본문 + 검증표 + audit 메모리 + 영상·대본 + 운영 문서).

### 금지 패턴 — 거짓 19 건 원인

| 패턴 | 설명 | 해결 |
|---|---|---|
| A | 분기값을 연도 라벨로 표기 | 표 헤더에 `(Q4 스냅샷)` 또는 `(1년치 합산)` 명시 |
| B | 비율 분기값과 1 년치 marginWaterfall 같은 표에 | 연간 GP 마진은 별도 표 |
| C | `capitalOverview` 소계를 총합으로 인용 | `notesDetail.borrowings` "합계" 행 사용 |
| D | dartlab 출력을 풀어 쓰면서 단위 변환 실수 | 절대로 다른 숫자 넣지 않음 |
| E | 검증 안 된 외부 출처 수치 | 검증표에 "외부 인용" 명시 + 출처 링크 |

### scan 숫자 — CFS 로 재확인한다

`scan` "OI YoY +400%" · "OPM 7.3%" 같은 급등·고마진은 **별도(OFS) 기준** 일 수 있음. **반드시 CFS(연결) 실측으로 재확인**.

사례: 현대코퍼레이션 `scan` OPM 7.3% → CFS 실측 1.85%.

### scan YoY 급등 — 기저효과 의심한다

"+400%" = 전년이 거의 0 이었다는 뜻. 분기 단발성일 수 있음. 연간 3~5 년 시계열을 본 뒤에만 "턴어라운드·급등" 프레임 사용.

## 6. 자산 정책 — SVG · 이미지 · 파일명

자산은 각 포스트 폴더의 `assets/` 에서 관리. 빌드 시 `/blog/assets/*` URL 로 평탄 복사.

### 위치 · 파일명

- 자산은 반드시 `assets/` 안에. 루트 `blog/assets/` 쓰지 않음.
- 본문 참조는 항상 `./assets/...` 상대경로.
- 모든 자산 파일명은 **포스트 번호 접두사**: `012-depreciation-flow-map.svg`.

**반복 실패** — 다른 포스트와 같은 파일명 = 빌드 실패.

### SVG 시스템 — 최소 5개 필수, 역할이 분명하게

> **⛔ 하드 게이트: SVG 5개 미만 = 발행 불가.** 인라인 차트(`ComboChart`, `BarChart`)는 SVG 카운트에 포함하지 않는다. `assets/` 폴더의 `.svg` 파일이 5개 이상이어야 한다.

| 역할 | 설명 |
|---|---|
| hero-map | 글의 질문과 답을 한 화면에 압축 |
| decision-tree | 언제 무엇을 먼저 볼지 분기 |
| comparison-matrix | 좋은 경우 vs 위험한 경우 |
| timeline-flow | 전개 순서·파이프라인·읽기 순서 |
| signal-card | 경고 신호·자주 놓치는 패턴 |
| checklist-board | 마지막 행동 요약 |
| evidence-layer | 본문 근거를 다시 정리 |

**반복 실패** — 같은 역할 SVG 2 개 이상 반복.

### 캔버스

| 용도 | 사이즈 |
|---|---|
| 넓은 흐름도 | 1200×720 |
| 비교표·매트릭스 | 1200×820 |
| 체크리스트·카드형 | 1080×720 |
| 모바일 친화 소형 | 960×640 |

`800x430` 하나만 반복하지 않는다.

### 정보 밀도 · 시각 품질

- 첫 SVG 는 제목만 예쁜 그림이 아니라 **최소 1 개의 핵심 판단 문장** 포함.
- 정보형 SVG 는 **실질 라벨 6 개 이상**.
- 체크리스트형은 **항목 5 개 이상**.
- 색은 강조용. 정보 구분은 위치·레이어·라벨로.
- 박스만 늘어놓지 않는다 (흐름·연결선·강조 배지·요약 문장 함께).
- 한국어 본문이면 SVG 도 한국어 우선 (DART · EDGAR · 10-K 같은 공식 표기 유지).
- 모바일 360px 폭에서도 첫 줄 질문과 핵심 라벨 읽혀야 함.

**반복 실패** — 텍스트 3 개 이하 = 장식 → 우선 교체. XML 파서 오류 / 텍스트 노드 ≤3 / 모든 글 같은 박스 템플릿 / 첫 SVG 가 글의 질문 설명 못 함 / 본문과 역할 겹침.

### 접근성 · XML

- SVG 안의 `&`, `<`, `>` 반드시 XML escape.
- `<title>` · `<desc>` 권장.
- alt 는 설명형으로.

### 유튜브 영상 임베딩

글에 대응 영상이 있으면 카테고리 상관없이 **본문 최상단** (frontmatter 바로 아래, metadata 인용 블록 앞) 에 `<YouTube>` 컴포넌트로 삽입.

1. **frontmatter**: `youtubeId: "WzBUtnbe_x0"` — `https://youtu.be/{ID}` 의 ID 부분만.
2. **script import**: `import YouTube from '$lib/components/YouTube.svelte';`.
3. **본문 배치**: `---` 구분자 + `<script>` 블록 다음, `> **{storyTemplate}** | ...` 메타 인용 앞.

```mdsvex
<script>
import YouTube from '$lib/components/YouTube.svelte';
...
</script>

<YouTube id="WzBUtnbe_x0" title="{회사명} — {핵심 후킹}" />

> **{storyTemplate}** | {업종} | {date} dartlab 실측
```

- `title` = 블로그 제목과 동일 한 줄.
- 영상 없는 글은 `youtubeId: ""` 유지, `<YouTube>` 삽입 생략.
- 레퍼런스: [SK하이닉스](blog/05-company-reports/01-000660-skhynix/index.md).

### 유튜브 설명란 — 영상 있는 글 전체

영상이 있는 글은 카테고리 상관없이 **유튜브 설명란도 블로그 품질로 작성**한다.

**설명란 구조 (순서 고정)**:

```
[도입 — 3~5줄]
블로그 관통선을 압축한 서사. 핵심 숫자 3~5개를 인과로 엮어서
"이 영상 왜 봐야 하는지"를 설명란만 읽어도 알게 한다.

[중단 — 영상 핵심 논점 3~5개]
블로그 각 막의 핵심을 한 줄씩.
각 줄이 "그래서 뭐?" 에 답해야 한다.
예: "매출원가율 141.6% — 100원어치 팔면 141원이 나간다"

📊 상세 분석 블로그:
{블로그 URL}

dartlab:
https://eddmpython.github.io/dartlab/
```

**품질 기준**:
- 설명란 자체가 **영상의 축약본** — 읽기만 해도 핵심 논점 전달.
- 블로그 관통선이 설명란에도 관통해야 함.
- 숫자는 반드시 인과와 함께 ("전기요금 1 원 = 5,600 억" ✓, "매출 97 조" ❌).

### 썸네일 원본 배경 보관

**FLUX 로 생성한 썸네일 원본 배경은 항상 `blog/{category}/{folder}/assets/{NN}-thumbnail-bg.webp` 로 보관한다.** 이게 없으면 레이아웃을 바꿔야 할 때마다 FLUX 재호출로 돈·시간 낭비.

- 최종 합성 썸네일: `landing/static/thumbnails/{code}-{slug}.webp` (덮어쓰기 OK).
- 원본 FLUX 배경: `blog/.../assets/{NN}-thumbnail-bg.webp` (덮어쓰기 없음).
- `gen_thumbnails.py` 같은 재생성 스크립트는 **반드시 assets/ 의 원본 배경을 읽어서** 합성.
- 원본 배경 파일명은 `{NN}-thumbnail-bg.webp` 고정.

**Phase 3 체크리스트 (썸네일 생성 시)**:
1. FLUX 로 배경 생성 → `assets/{NN}-thumbnail-bg.webp` 로 먼저 저장.
2. `gen_thumbnails.py` 스타일로 합성 → `landing/static/thumbnails/{code}-{slug}.webp`.
3. 둘 다 커밋. 원본 배경 커밋 누락 시 차단.

**반복 실패** — 2026-04-16 META·TSLA·하이브·IONQ 사고 직접 원인. MNST(36) 는 원본 남겨 재생성 가능, 37~40 은 원본 없어서 FLUX 재호출 필요.

### 글 단위 SEO 체크리스트 12 개

1. **제목 ≤60 자** — 검색형 질문 또는 판단 문장. "전자공시" 키워드 자연 포함 권장 (네이버).
2. **description ≤160 자** — 글 핵심 결론 한 줄, 첫 2 문단과 일관.
3. **slug** — 영문 케밥, 회사명 + 한두 단어 (예: `01-000660-skhynix` 폴더, URL `/blog/skhynix`).
4. **frontmatter `tags`** — 검색·필터링용 5~8 개.
5. **본문 H2 구조** — H1 핵심 질문에 단계적으로 답. 영문 H2 금지. 5~8 개 권장. **H2 자체가 GEO·SGE 인용 단위.**
6. **첫 2 문단 (200~300 자)** — 핵심 답변 들어가야.
7. **내부 링크 ≥3** — 같은 카테고리·시리즈·회사, 시리즈 정의 페이지.
8. **외부 출처 ≥3** — 보도·공식문서·위키. 본문 직접 링크.
9. **이미지 alt** — 단순 설명 없이. **figcaption 첫 줄과 동일 본문 한 줄**. 한글 OK.
10. **표 캡션 · 코드블록** — 표 위 2~4 문장 한 단락 + 표 아래 한 줄 풀이. 코드블록은 dartlab 호출 그대로.
11. **단락당 한 사실** — AI 가 인용할 때 단락 단위로 자름. 200 자+ 단락은 둘로.
