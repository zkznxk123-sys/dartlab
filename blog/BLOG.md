## 1. 정체성 — 기업이야기꾼

dartlab 블로그는 **재무제표의 "왜?"를 풀어내는 엔진**이다.

재무제표를 열었더니 이상한 게 보이고, 파고 들어가면 **"아 그래서"** 하는 발견의 쾌감. "이 회사 부채 37조인데 빚이 0원" / "매출 1.6조인데 점유율 52%" / "영업이익률 4%→18%, 공장 안 지었는데 왜?" — 이게 블로그의 정체성이다.

**핵심 약속**:
- 공시 원문을 직접 읽고 판단하는 **실전 교재**
- dartlab 없이도 가치 있는 글이어야 함
- 숫자는 **정직성 최우선**
- 독자가 "나도 재무제표 열어보고 싶다"고 느끼면 성공

## 3. 제작 파이프라인 — 5에이전트 체인

모든 기업이야기는 단일 파이프라인. 단독 작업 금지.

### Phase 0: 데이터 탐색

dartlab **5엔진을 직접 돌려** "이상한 것, 재미있는 것, 모순"을 찾는다. **사람이 아는 이야기가 아니라 데이터가 말하는 이야기**를 먼저 찾는 게 핵심.

```python
# 1) scan — 동종사 상대 위치
dartlab.scan("profitability"); dartlab.scan("growth"); dartlab.scan("quality")
dartlab.scan("capital"); dartlab.scan("debt"); c.governance()

# 2) analysis — 14축 분해
c.analysis("수익구조"); c.analysis("비용구조"); c.analysis("이익품질")
c.analysis("자본배분"); c.analysis("자금조달"); c.analysis("현금흐름")

# 3) quant — 시장 평가
c.quant("종합"); c.quant("팩터")
```

→ **"이 회사에서 발견한 이상한 것 5~10개"** 리스트 → 관통선 후보 선택.

**SEO 경쟁 분석**: "회사명 + 분석" 네이버 상위 5개 글의 H2 비교 → 내 글에 없는 관점 목록화.

### Phase 1: 기획 (4에이전트 병렬)

| 에이전트 | 역할 | 산출물 |
|---|---|---|
| **기획자** | 핵심 질문 1개, 관통선, 막 구조 | 막 구조표 |
| **재무분석가** | dartlab 실행, "이상한 숫자" 5개 발굴 | 막별 테이블 |
| **스토리작가** | 구체적 장면, 인물 대사, 리듬 | 장면 5개 |
| **마케터** | 제목 5개, description 3개 | 최종 선택지 |

#### [최우선] 관통선 단일 인과 규칙

**90점 이상의 글은 관통선이 "하나의 인과"로 1막부터 끝까지 모든 막을 꿰뚫는다.**

- 관통선 = **"왜 A인데 B인가?"** 형태의 질문 1개
- **병렬 서사 금지**: "세 겹의 재무제표" 같은 구조는 85점 천장
- 매 막의 끝이 다음 막의 원인
- 관통선 무관 토픽은 별도 막 X, **관련 막에 1~2문단으로 녹임**
- 검증: 모든 막의 제목 나열 후 "이 막을 빼면 관통선이 끊기는가?" → 끊기지 않으면 삭제/흡수

**사례**:
- ✅ SK하이닉스: "5번 죽을 뻔한 회사가 왜 매번 더 강해졌는가"
- ✅ 대한조선: "조선소인데 OPM 24% 가 어떻게 가능한가"
- ❌ 현대글로비스: "세 겹의 재무제표" → 병렬 → 85점 천장

### Phase 2: 집필 (재무분석가 → 스토리작가 순차)

- 매 막을 **재무분석가(데이터+해석) → 스토리작가(장면+리듬)** 순서
- 매 막 "왜?"로 시작, **관통선 질문의 답을 향해** 흐르기
- **매 막 끝에 다음 막으로의 인과 다리 1문장 필수**

#### 막 개수 유연성

- **6막은 기본 템플릿, 고정 아님.** 이야기거리가 많으면 7막, 8막까지 확장 가능.
- 확장 기준: 막 하나가 3,500자 넘어가면 두 막으로 쪼개기, 관통선이 흐려지면 합치기.
- "막 개수 맞추느라 이야기 잘라내기" 금지. 이야기가 요구하는 구조가 우선.

#### [필수] 과거~현재 패턴 + 산업 패턴 + 투자 포인트 막

**모든 기업이야기 회사편에 반드시 1막 포함한다.** 위치는 자연스러운 곳(마지막 직전 또는 6막 내부 섹션). 내용 3가지:

1. **이 종목의 과거~현재 패턴** — 이번 사이클이 처음인지 반복인지, 과거 유사 국면에서 어떻게 돌아왔는지. 9년 시계열에서 반복 패턴을 추출.
2. **산업 패턴** — 소속 산업의 사이클, 경쟁 구조, 변곡점. 한국 종목은 `c.industry` 엔진 활용 가능. US 종목은 10-K Risk Factors + 산업 보고서 인용.
3. **투자 포인트** — 이 종목을 보는 사람이 다음 분기·1년에 봐야 할 체크포인트 3~5개. "로보택시 분기 매출", "CAPEX/매출 비율", "유효세율" 같은 구체 지표.

⚠ **현재 `industry` 엔진은 한국 전용.** EDGAR 종목(META/TSLA 등)은 외부 출처 + 질적 분석으로 대체.

#### [권장] analysis 서사 구조 활용

analysis 엔진은 **"이야기가 숫자를 만들고 숫자가 이야기를 검증한다"** 사상으로 재편됐다 (Damodaran 생애주기 + 스토리 일관성 검증 + `review.narrative` 7가지 인과 패턴). 블로그 작성 시 개선 가능성 발견 → 제안 후 반영:

- `c.analysis("생애주기")` — 고성장/성숙/쇠퇴 판정과 modelHint → 관통선 프레이밍에 활용
- `review.narrative` NarrativeThread — BS/IS/CF 간 자동 감지된 7가지 인과 패턴 → 막 간 인과 다리 문장에 활용
- `summaryFlags` — 종합평가 플래그 → 6막 검증 체크리스트로 활용
- 스토리-숫자 일관성 검증 실패 시 → 관통선 재검토 신호

기존 글 보강 시에도 위 엔진 출력과 대조해 "글이 주장하는 이야기"와 "엔진이 숫자로 말하는 이야기"가 충돌하면 글 우선 수정.

### Phase 3: SVG + 이미지 + 썸네일

**SVG 규칙 (5~7개)**: 다크 `#0a0e1a`, 폰트 `-apple-system, 'Segoe UI', sans-serif`, 타이틀 한국어 필수. 텍스트 `fill=#f1f5f9`, 부제 `#64748b`, 강조 `#fbbf24`. A등급 `#22c55e`, C/D `#f59e0b/#ef4444`.

**이미지 생성 (FLUX, Replicate 전용)**: 2~3개, `black-forest-labs/flux-1.1-pro`, 영문 프롬프트 + "no text", 사실적 장면, 로고 금지. 저장: `assets/{nn}-{slug}.webp`. **다른 provider 금지** (memory feedback 약속 흡수).

**썸네일 (og:image 1200×630)**: 좌측 다크 그라데이션 + 회사명 + 제목 2줄, 우측 FLUX 배경, `malgunbd.ttf`. 저장: `/thumbnails/{stockCode}-{slug}.webp`.

**Phase 3 실행 순서**: SVG 5~7 Write → FLUX 2~3 (12초 간격) → 본문 참조 → SEO audit → **이미지 0개면 안 됨** → FLUX 완료 후 커밋.

## 4. 품질 기준 — 5가지 기본 약속

1. **제목이 후킹이다** (60자 이하, 검색형 질문 또는 판단 문장)
   - ✅ "SK하이닉스 (000660) — 한국 반도체 30년의 미스터리, 그리고 영업이익률 58%"
   - ❌ "SK하이닉스 분석"

2. **첫 2문단 안에 핵심 답** (200~300자) → Featured Snippet + LLM 인용 직결
   - 도입부 관성 금지. "이상한 재무제표" 프레임 최대 1번
   - 매 글 도입부는 **그 기업만의 고유한 장면/숫자/질문**

3. **단일 핵심 질문이 글 전체 지탱**

4. **고유 H2 구조** — 템플릿 복사 금지

5. **닫힘성** — 이 글에서 할 말을 이 글에서 끝낸다
   - 금지: "다음 글에서", "심화 글 예고"
   - 존재 안 하는 글 링크는 신뢰 깎음

| **숫자 자랑** "14축 체계", "19개 분석", "6막 구조" | 광고처럼 보임 | "수익성, 안정성, 현금흐름 등 시장 전체를 읽는다" |

| **"무료" 표현** | provider/API 가격 언급 자체 금지 | 가격 표현 0건 |
| **"다음 글에서 다룬다"** | 존재 안 하는 글 링크 신뢰 깎음 | 이 글에서 끝낸다 |

### 각 막의 구조 규칙

- **매 막 "왜?"로 시작**
- **표 앞에 맥락 1줄 필수** — 숫자 먼저 금지
- **막 전환 시 인과 연결 1문장 필수** — "3막의 결과가 4막을 가능하게 한 것이다"
- **마지막 막은 판단으로 끝나기** — 체크리스트/나열 금지
- **각 막에 H3 소제목 3~5개** — 글 전체 H3 합계 15개 이상 목표
  - 형식: `### 매출원가율 141.6% — 팔수록 적자` (숫자 + 한 줄 해석)
- **중복 막 금지**
- **각 막에 dartlab 코드 + 표 1개 이상**

좋은 H3 예: LG이노텍 23, 한온시스템 18, HDC현산 26.
나쁜 예: SK하이닉스 3, 삼양식품 3 (보강 필수).

### 용어 풀어쓰기 (강제)

독자는 재무제표를 처음 보는 사람. 전문 용어는 **같은 문장 안에서** 풀어쓰기.

| 쓰지 말 것 | 이렇게 바꿔라 |
|---|---|
| OPM 6.2% | 영업이익률 6.2% (매출 대비 영업이익 비율) |
| GP 마진 44% | 매출총이익률 44% (물건 팔고 원가 빼면 44% 남는다) |
| OCF | 영업활동현금흐름 (실제 장사해서 들어온 현금) |
| FCF | 잉여현금흐름 (영업현금에서 투자비를 뺀 진짜 남는 돈) |
| CCC -55일 | 현금전환주기 -55일 (물건 팔기 전에 돈이 먼저 들어오는 구조) |
| ICR 0.92x | 이자보상배율 0.92배 (번 돈으로 이자도 못 갚는 상태) |
| ROIC / ROE | 투하자본수익률 / 자기자본수익률 |
| 운전자본 | 영업에 묶여있는 돈 (재고+매출채권-매입채무) |
| 감가상각 | 과거에 산 설비 값을 매년 조금씩 비용으로 깎는 것 |

**규칙**: 괄호 풀이 1줄 필수, 첫 등장 시 풀이 후 이후 약어 OK, 비유 권장.

## 5. dartlab 호출 방식 — 8가지 패턴 + 데이터 단위

블로그에 자주 등장하는 호출. 본문 코드블록에 그대로 박는다 — **독자가 복사 → 실행 → 검증 가능**해야.

### 1) IS 9년 시계열 (1년 합산)

```python
import dartlab
c = dartlab.Company("000660")
c.select("IS", ["매출액","매출원가","매출총이익","판매비와관리비","영업이익","당기순이익"])
# 분기 컬럼 반환 → 1년치 합산 필요
df = c.select("IS", ["매출액"])
row = df.row(0, named=True)
annual = {y: sum(row.get(f"{y}Q{q}", 0) or 0 for q in range(1, 5)) for y in range(2017, 2026)}
```
⚠ **분기 단독값과 1년 합산을 절대 섞지 말 것**

### 2) BS Q4 스냅샷

```python
c.select("BS", ["자산총계","부채총계","자본총계","현금및현금성자산"])
# BS는 시점 데이터 → 합산 X, 각 연도 Q4만
```

### 3) CF 9년 시계열 + 잉여현금 파생행

```python
c.select("CF", ["영업활동현금흐름","유형자산의 취득","배당금지급"])
# CF − CAPEX = 잉여현금 — 본문 표에 반드시 포함
```

### 4) 비율 분기 시계열

```python
c.select("ratios", ["영업이익률 (%)","매출총이익률 (%)","부채비율 (%)"])
```
⚠ `c.analysis("financial","수익성")["marginWaterfall"]` 는 1년치. 섞지 말 것.

### 5) 수익성 분석

```python
prof = c.analysis("financial", "수익성")
prof["marginWaterfall"]["history"][0]      # 1년치 마진 분해
prof["roicTree"]["history"][0]             # ROIC 트리
prof["penmanDecomposition"]["history"]     # Penman RNOA/FLEV/NBC/Spread
```
**marginDriver 는 dartlab 자체 분류이므로 코드블록으로 명시 인용 가능**:
```
"marginDriver": "높은 가격결정력 (매출총이익률 > 40%)"
```

### 6) 자금조달

```python
fund = c.analysis("financial", "자금조달")
fund["capitalOverview"]   # 표시용 요약 (총자산/총부채/자기자본/순차입금만)
fund["fundingSources"]["notesDetail"]["borrowings"]  # 차입금 항목별
```
⚠ **capitalOverview 의 "순차입금" 은 내부 정의. "총 차입금" 필요하면 `notesDetail.borrowings` 의 "합계" 행 사용.**
사례: SK하이닉스 본문 "차입금 8.16조" 는 유동성 소계. 진짜 22.25조.

### 7) 자본배분 — 배당 (두 출처 차이)

```python
ca = c.analysis("financial", "자본배분")
ca["dividendPolicy"]["history"]  # CF dividendsPaid 1년치 합산

c.report.dividend.dps  # 주당 현금배당금 (DART 정기보고서)
```
⚠ 두 출처는 다를 수 있음 (CF=자본 재구성/우선주 포함, report=보통주만). 어느 출처를 썼는지 검증표에 명시.

### 8) 신용등급

```python
cr = c.credit("등급")
cr["grade"]; cr["score"]; cr["healthScore"]; cr["divergenceExplanation"]
```

### 분기 vs 연간 vs Q4 — 표 헤더 단일 라벨

| 데이터 종류 | 단위 | 표 헤더 라벨 |
|---|---|---|
| IS (매출/원가/이익) | 1년치 합산 | `(1년치 합산, 조원)` |
| BS (자산/부채/자본) | 시점 스냅샷 | `(Q4 스냅샷, 조원)` |
| CF (현금흐름) | 1년치 합산 | `(1년치 합산, 조원)` |
| 비율 | 분기 | `(분기, %)` |
| marginWaterfall | 1년치 | `(1년치 분해, %)` |

**한 표 안에 여러 단위를 절대 섞지 않는다.**

### 회사 종합편 필수 6가지 데이터

| # | 데이터 | 호출 | 본문 위치 |
|---|---|---|---|
| 1 | IS 9년 시계열 표 | `c.select("IS", [...])` 분기 합산 | 수익성/마진 막 시작 |
| 2 | BS 9년 Q4 스냅샷 | `c.select("BS", [...])` Q4만 | 안정성/자금조달 막 시작 |
| 3 | CF 9년 + 잉여현금 파생행 | `c.select("CF", [...])` 분기 합산 | 현금/자본배분 막 시작 |
| 4 | 핵심 비율 분기 시계열 1~3개 | `c.select("ratios", [...])` | 핵심 변곡점 막 |
| 5 | dartlab 자체 분석 인용 ≥1 | marginDriver/flag/divergence | 메커니즘 막 |
| 6 | 신용등급 카드 | `c.credit("등급")` | 안정성 막 |

### 표 작성 규칙

1. **컬럼 = 시점, 좌측이 가장 최신** (2025 → 2024 → ... → 2017)
2. **첫 컬럼 = 항목명**, 짧고 명확하게
3. **단위는 표 헤더에 명시**
4. **숫자 컬럼 우측 정렬** (`---:`)
5. **강조 `**굵게**`** — 정점/바닥/사상 최대
6. **표 아래 한 줄 풀이** — `표시: **+58.40** = 사상 최대`

### 단위 결정

모든 행이 0.5~1000 범위에 들어가는 단위.

| 회사 | 단위 |
|---|---|
| 대형 (SK하이닉스, 삼성전자) | 조원 |
| 중형 (삼양식품) | 억원 |
| 비율 | % |

같은 글에서 단위 혼용 OK (표마다 헤더에 명시).

### 수치 정직성 6원칙 (가장 중요)

지난 라운드 19건 거짓의 80% 가 다음 6가지 중 하나를 어겼다.

1. **분기 vs 연간 라벨을 절대 섞지 않는다**
   - ❌ `| 연도 (Q4) | 매출 |` ← Q4 단독값인데 "2025" 로 라벨 (1년치로 오해)
   - ✅ `| 항목 (Q4 스냅샷, 조원) | 2025Q4 | 2024Q4 |`

2. **모든 인용 수치는 검증표에 등록**
   ```markdown
   | 본문 수치 | dartlab 호출 | 결과 |
   |---|---|---|
   | 2025 연간 매출 97.15조 | `c.select("IS",["매출액"])` 분기 합산 | ✅ 실측 |
   ```
   검증표에 없는 숫자가 본문에 있으면 발행 차단.

3. **핵심 후킹 수치는 두 번 검증** (작성 시 + 발행 직전 dartlab 재실행)

4. **dartlab 자체 출력은 직접 인용 가능하지만 인용임을 명시** (코드블록 또는 따옴표)

5. **"약 X조" / "거의 Y배" 어림 표현은 정확값 옆에만**
   - ✅ "27.52조 (2018 정점의 약 1.7배)"
   - ❌ "약 30조 정도가 1차 정점의 두 배에 가깝다"

6. **거짓 발견 시 5곳 같이 갱신** (본문 + 검증표 + audit 메모리 + 영상/대본 + 운영 문서)

### 금지 패턴 (거짓 19건 원인)

| 패턴 | 설명 | 해결 |
|---|---|---|
| A | 분기값을 연도 라벨로 표기 | 표 헤더에 `(Q4 스냅샷)` 또는 `(1년치 합산)` 명시 |
| B | 비율 분기값과 1년치 marginWaterfall 같은 표에 | 연간 GP 마진은 별도 표 |
| C | capitalOverview 소계를 총합으로 인용 | `notesDetail.borrowings` "합계" 행 사용 |
| D | dartlab 출력을 풀어 쓰면서 단위 변환 실수 | 절대로 다른 숫자 X |
| E | 검증 안 된 외부 출처 수치 | 검증표에 "외부 인용" 명시 + 출처 링크 |

### scan 숫자 맹신 금지 — OFS vs CFS 재확인

scan "OI YoY +400%", "OPM 7.3%" 같은 급등·고마진은 **별도(OFS) 기준**일 수 있음.
**반드시 CFS(연결) 실측으로 재확인**.
사례: 현대코퍼레이션 scan OPM 7.3% → CFS 실측 1.85%.

### scan YoY 급등은 기저효과 의심

"+400%" = 전년이 거의 0이었다는 뜻. 분기 단발성일 수 있음.
연간 3~5년 시계열을 본 뒤에만 "턴어라운드/급등" 프레임 사용.

### [최우선] 글 말미 3섹션 독립 H2 필수 구조

**모든 회사 종합편 글 말미는 반드시 3개 독립 H2 블록으로 분리.** 부록 아래 서브섹션으로 숨기면 안 됨. 순서 고정.

```markdown
## 검증표
본문 모든 인용 수치 → dartlab 호출 키 → 결과 행. 본문의 숫자 중 이 표에 없는 건 발행 차단.

## 공시 / Filings
DART 정기보고서 · EDGAR 10-K / 10-Q / 8-K · IR 발표자료 · 감사보고서 직접 링크. 원문 URL 필수 (인용만 하는 보도 URL X).

## 재무제표 — 최근 5개년
IS / BS / CF 3표, 각 5년, **단위 통일**. "—" 빈칸 금지, 추정값 금지, dartlab 실측만.
```

**금지 패턴**: `## 부록: 재무제표 5년` 아래 `### 검증표` 서브로 묶기, 공시 섹션 생략, 외부 언론 인용으로 공시 섹션 대체.

**이유**: 검증표와 공시 링크는 글의 신뢰 근간 — 독자가 한눈에 찾아 역추적할 수 있어야 한다. 부록에 섞어 넣으면 "숨겼다" 는 신호. 2026-04-16 META/TSLA/하이브/IONQ 4건 회귀 발견.

### 부록 재무제표 5년 표 필수

위 `## 재무제표 — 최근 5개년` 블록이 이 역할. 단위 통일, "—" 빈칸 금지, 추정값 금지, dartlab 실측만.

### 데이터 갱신 시점

- 글 제목 옆에 "데이터 기준: YYYY-MM-DD dartlab" 명시
- 검증표에 "📅 dartlab 실측 YYYY-MM-DD"
- 큰 분기 변동 후 핵심 후킹 한 번 더 검증

## 6. 자산 정책 — SVG / 이미지 / 파일명

자산은 각 포스트 폴더의 `assets/` 에서 관리. 빌드 시 `/blog/assets/*` URL 로 평탄 복사.

### 위치 + 파일명

- 자산은 반드시 `assets/` 안에. 루트 `blog/assets/` 사용 X
- 본문 참조는 항상 `./assets/...` 상대경로
- 모든 자산 파일명은 **포스트 번호 접두사**: `012-depreciation-flow-map.svg`
- 다른 포스트와 같은 파일명 = 빌드 실패

### SVG 시스템 — 5~7개, 역할이 분명해야

| 역할 | 설명 |
|---|---|
| hero-map | 글의 질문과 답을 한 화면에 압축 |
| decision-tree | 언제 무엇을 먼저 볼지 분기 |
| comparison-matrix | 좋은 경우 vs 위험한 경우 |
| timeline-flow | 전개 순서, 파이프라인, 읽기 순서 |
| signal-card | 경고 신호, 자주 놓치는 패턴 |
| checklist-board | 마지막 행동 요약 |
| evidence-layer | 본문 근거를 다시 정리 |

같은 역할 SVG 2개 이상 반복 X.

### 캔버스

| 용도 | 사이즈 |
|---|---|
| 넓은 흐름도 | 1200x720 |
| 비교표/매트릭스 | 1200x820 |
| 체크리스트/카드형 | 1080x720 |
| 모바일 친화 소형 | 960x640 |

`800x430` 하나만 반복 X.

### 정보 밀도 + 시각 품질

- 첫 SVG 는 제목만 예쁜 그림이 아니라 **최소 1개의 핵심 판단 문장 포함**
- 정보형 SVG 는 **실질 라벨 6개 이상**
- 체크리스트형은 **항목 5개 이상**
- 텍스트 3개 이하 = 장식 → 우선 교체
- 색은 강조용. 정보 구분은 위치/레이어/라벨로
- 박스만 늘어놓지 말기 (흐름·연결선·강조 배지·요약 문장 함께)
- 한국어 본문이면 SVG 도 한국어 우선 (DART/EDGAR/10-K 같은 공식 표기 유지)
- 모바일 360px 폭에서도 첫 줄 질문과 핵심 라벨 읽혀야 함

### 접근성 / XML

- SVG 안의 `&`, `<`, `>` 반드시 XML escape
- `<title>` `<desc>` 권장
- alt 는 설명형으로

### 리라이트 우선

- XML 파서 오류 / 텍스트 노드 ≤3 / 모든 글 같은 박스 템플릿 / 첫 SVG 가 글의 질문 설명 못 함 / 본문과 역할 겹침

### 유튜브 영상 임베딩 (회사분석보고서만)

회사 글에 대응 영상이 있으면 **본문 최상단**(frontmatter 바로 아래, metadata 인용 블록 앞)에 `<YouTube>` 컴포넌트로 삽입.

### 썸네일 (og:image) — 회사분석보고서만

**[최우선] 풀블리드 이미지 + 좌측 그라데이션 + 흰 제목 오버레이 (MNST 스타일 고정)**

MNST(36) 이후 썸네일이 2분할(좌 텍스트 박스 / 우 이미지)로 회귀해서 일관성이 깨졌다. **다시는 2분할 금지**. 모든 썸네일은 이미지가 캔버스 전체(1200×630)를 덮고, 그 위에 어두운 그라데이션 + 흰 제목 텍스트가 **오버레이**된다.

- 배경: FLUX 이미지가 캔버스 전체 채움 (비율 유지 crop, 좌우 텍스트 박스 없음)
- 전체 어두운 필터 `(10,14,26,70)` + 좌측 가로 그라데이션 `alpha = 200*(1-x/900)`
- 합성: Pillow. 레퍼런스 스크립트 [gen_thumbnails.py](scripts/blog/gen_thumbnails.py)
- 사이즈: 1200×630, WebP (quality 90)
- 텍스트 레이어:
  - 좌상단 `{회사명 (종목코드)}` malgun.ttf 24px `#94a3b8`
  - 우상단 `dartlab` malgunbd.ttf 22px `#f1f5f9`
  - 중앙 좌측 제목 2줄 malgunbd.ttf 58px 흰색, y=190부터 80px 간격
  - 제목 아래 부제 1줄 malgun.ttf 22px `#94a3b8`
  - 우하단 `avatar-chart.png` 160×160
- 저장 경로: `landing/static/thumbnails/{code}-{slug}.webp` 1곳만
- frontmatter: `thumbnail: /avatar-chart.png` (리스트 아바타), `ogImage: /thumbnails/{code}-{slug}.webp`

**금지 패턴**:
- 좌측 반 = 텍스트 박스 / 우측 반 = 이미지 의 2분할 레이아웃
- 이미지가 캔버스 일부만 차지하고 여백에 단색 박스 깔린 형태
- 제목 텍스트가 별도의 반투명 박스/카드에 들어간 형태

**참조 기준작**: MNST(36). 현재 META(37)/TSLA(38)/하이브(39)/IONQ(40) 4건은 이 규칙 위반이라 재생성 대기.

### [최우선] 썸네일 원본 배경 보관 규칙

**FLUX로 생성한 썸네일 원본 배경은 항상 `blog/{category}/{folder}/assets/{NN}-thumbnail-bg.webp` 로 보관한다.** 이게 없으면 레이아웃을 바꿔야 할 때마다 FLUX 재호출로 돈·시간 낭비.

- 최종 합성 썸네일: `landing/static/thumbnails/{code}-{slug}.webp` (덮어쓰기 OK)
- 원본 FLUX 배경: `blog/.../assets/{NN}-thumbnail-bg.webp` (덮어쓰기 금지)
- `gen_thumbnails.py` 같은 재생성 스크립트는 **반드시 assets/ 의 원본 배경을 읽어서** 합성
- 원본 배경 파일명은 반드시 `{NN}-thumbnail-bg.webp` (다른 이름 금지)

**원본이 없는 글은 썸네일 재생성 불가능** → 2026-04-16 META/TSLA/하이브/IONQ 사고 직접 원인. MNST(36)는 원본을 남겨 재생성 가능, 37~40 은 원본 없어서 FLUX 재호출 필요.

**Phase 3 체크리스트 (썸네일 생성 시)**:
1. FLUX 로 배경 생성 → `assets/{NN}-thumbnail-bg.webp` 로 먼저 저장
2. `gen_thumbnails.py` 스타일로 합성 → `landing/static/thumbnails/{code}-{slug}.webp`
3. 둘 다 커밋. 원본 배경 커밋 누락 시 차단.

### 검증 체크리스트

1. 본문 자산 경로가 `./assets/...` 인지
2. 파일명 접두사가 포스트 번호와 일치
3. `uv run python -X utf8 scripts/auditBlog.py` XML/밀도 이슈 확인
4. `npm run build` 후 `landing/build/blog/assets` 복사 확인
5. 자산 파일명 충돌 0
6. 모바일에서도 핵심 라벨 읽힘
