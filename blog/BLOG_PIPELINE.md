# 기업이야기 블로그 생산 파이프라인

HD현대일렉트릭·고려아연·에이피알에서 검증. 모든 기업이야기는 이 순서를 따른다.

---

## 1. 핵심 사상

**재미 = 분석의 "왜?"**
- 재무제표를 열었더니 이상한 게 보이고, 파고 들어가면 "아 그래서" 하는 **발견의 쾌감**
- "이 회사 부채 37조인데 빚이 0원" / "매출 1.6조인데 점유율 52%" / "영업이익률 4%→18%, 공장 안 지었는데 왜?"
- 독자가 "나도 재무제표 열어보고 싶다"고 느끼면 성공

**주객전도 금지**
- 주: 재무제표 숫자. 객: 역사/인물/사건
- 매 막이 "왜?"로 시작. 인물/사건은 답의 일부로만 등장
- 설명 톤("~이다") 금지 → 발견 톤("~인데 왜?", "뜯어보면")

**부채비율 과잉 강조 금지**
- 크리티컬 아니면 "위험" 톤 쓰지 말 것
- 영업부채가 큰 회사 = 구매력 있는 것

**도입부 관성 금지**
- "이상한 재무제표" "재무제표를 열면 이상한 게 보인다" 같은 공식 도입부 반복 금지
- 매 글의 도입부는 **그 기업만의 고유한 장면/숫자/질문**으로 시작해야 한다
- 한화에어로: "삼성이 무기를 버렸다" / 고려아연: "50년 만에 첫 순손실" / 에이피알: "화장품+가전" — 각각 다른 첫 문장
- "이상한 재무제표"를 도입부 프레임으로 쓰는 건 **최대 1번**. 이미 썼으므로 이후 금지
- 마케터 에이전트가 도입부 프레임 중복 여부를 체크

---

## 2. 5 에이전트 (기획 단계부터 전원 참여)

| 에이전트 | 역할 | 산출물 |
|----------|------|--------|
| **기획자** | 핵심 질문 1개, 관통선, 막 구조, 5엔진 교차 포인트 | 막 구조표 |
| **재무분석가** | dartlab 실행, 숫자 검증, "이상한 숫자" 5개 발굴 | 막별 테이블 + 해설 |
| **스토리작가** | 구체적 장면(날짜, 장소, 인물 대사), 문장 리듬 변주 | 장면 5개 + 리듬 가이드 |
| **마케터** | 제목, description, 핵심 한 줄, 마지막 문장, 공유성 | 제목 안 5개 + description 3개 |
| **독자** | 완성된 글을 처음 보는 일반 독자 시점으로 평가 | 점수 + 약한 막 + 개선 포인트 |

**단독 작업 금지.** 한 에이전트가 전부 하면 리포트 톤이 됨.

---

## 3. 5엔진 활용 — 다양한 관점

analysis만 쓰지 말고, 기업에 맞는 엔진 2~3개를 골라 교차 관점에서 "어?"를 만든다.

| 엔진 | 블로그 관점 | 호출 예시 |
|------|-----------|----------|
| **analysis** | 14축 재무 분해 (비용구조, 이익품질, 자본배분 등) | `c.analysis("비용구조")` |
| **scan** | 동종사 상대 위치 (백분위), 지배구조 점수 | `dartlab.scan("profitability")` |
| **quant** | 기술적 3분해, 재무-시장 괴리, 8 백테스트 | `c.quant("종합")` |
| **macro** | 사이클 국면, 금리/환율/원자재 민감도 | `dartlab.macro("사이클")` |
| **credit** | 독립 신용등급(dCR), 외부 신평사 비교 | `c.credit("등급")` |

기획 시 체크리스트:
- [ ] scan: 동종사 몇 위? 백분위 이상한가?
- [ ] quant: 재무 A인데 시장 약세? 괴리?
- [ ] analysis 비용구조: 마진 원인이 원가? 판관비? 추세?
- [ ] macro: 사이클 국면? 환율/원자재 민감?
- [ ] credit: 신용등급 변화?

## 4. 단계별 파이프라인

### Phase 0: 데이터 탐색 — dartlab 엔진으로 이야기거리 발굴

**기획 전에** dartlab 5엔진을 직접 돌려서 "이상한 것, 재미있는 것, 모순"을 찾는다. 사람이 아는 이야기(뉴스, 위키)가 아니라 **데이터가 말하는 이야기**를 먼저 찾는 것이 핵심.

**탐색 에이전트 병렬 투입:**

1. **scan 탐색** — 전종목 횡단분석으로 이 회사의 상대적 위치를 찾는다
   ```python
   dartlab.scan("profitability")   # 수익성 — 동종사 중 몇 위?
   dartlab.scan("growth")          # 성장성 — 이상 성장? 급감?
   dartlab.scan("quality")         # 이익 품질 — 현금 안 따라오는 회사?
   dartlab.scan("capital")         # 자본배분 — 환원형? 희석형?
   dartlab.scan("debt")            # 부채 — 위험 등급?
   c.governance()                  # 지배구조 — 점수 낮은가?
   c.workforce()                   # 인력 — 급여/매출 괴리?
   ```
   → "동종사 대비 이상치", "업종 내 1위/꼴등", "2축 교차에서 튀는 점" 찾기

2. **analysis 탐색** — 14축을 전부 돌려서 이상한 축 찾기
   ```python
   c.analysis("수익구조")    # 매출 구성에 이상한 게 있는가
   c.analysis("비용구조")    # 원가/판관비 추세가 이상한가
   c.analysis("이익품질")    # 발생액이 높은가, CF가 안 따라오는가
   c.analysis("자본배분")    # 배당/자사주/재투자 배분이 이상한가
   c.analysis("자금조달")    # 자금 원천이 급변했는가
   c.analysis("현금흐름")    # CF 패턴이 이상한가
   c.analysis("종합평가")    # 스코어카드에서 A와 F가 동시에 있는가
   ```

3. **quant 탐색** — 시장이 재무를 어떻게 평가하는가
   ```python
   c.quant("종합")     # 기술적 판단 (강세/약세)
   c.quant("팩터")     # FF5 팩터 노출
   ```
   → "재무 A인데 시장 약세? 괴리가 있는가?"

**산출물**: "이 회사에서 발견한 이상한 것 5~10개" 리스트. 이 중 **관통선이 될 수 있는 것**을 골라 Phase 1로 넘긴다.

**SEO 경쟁 분석** (Phase 0에 포함):
- "회사명 + 분석" 네이버 검색 상위 5개 글의 H2 비교
- 상위 글에 있는데 내 글에 없는 관점 목록화
- 상위 글 평균 글자수 vs 내 목표 글자수

**"dartlab scan 발견" 앵글**: scan에서 데이터로 먼저 발견한 회사는 도입부에 "scan이 이 회사를 발견했다"를 써서 dartlab의 가치를 자연스럽게 보여준다. 발견 경로를 다양하게 — scan("profitability"), scan("capital"), 이상치 탐지 등.

**부수 효과**: 탐색 과정에서 scan 엔진의 인사이트 품질이 부족하면 → scan 엔진 개선 이슈로 등록.

### Phase 1: 기획 (4에이전트 병렬)
1. 기획자: **Phase 0 탐색 결과를 기반으로** 관통선 1개 확정, 막 구조 초안
2. 재무분석가: dartlab 실행 → "이상한 숫자 5개" (Phase 0에서 발견한 것 + 추가)
3. 스토리작가: 외부 리서치 → 인물/사건/장면 소재
4. 마케터: 제목 안 5개 + description 3개 + 핵심 한 줄 2개

### [최우선] 관통선 단일 인과 규칙 (2026-04-11 현대글로비스 교훈)

**90점 이상의 글은 관통선이 "하나의 인과"로 1막부터 마지막 막까지 모든 막을 꿰뚫는다.**

- 관통선은 **"왜 A인데 B인가?"** 형태의 질문 1개. 모든 막이 이 질문의 답을 향해 흘러야 한다
- **병렬 서사 금지**: "세 겹의 재무제표", "3가지 엔진" 같은 구조는 인과가 분산돼 85점 천장이 생긴다
- 매 막의 끝이 다음 막의 원인이 되어야 한다: "1막의 결과가 2막을 가능하게 했다"
- 관통선과 무관한 토픽(보스턴다이내믹스, 지배구조 등)은 **별도 막을 만들지 않고** 관련 막에 1~2문단으로 녹인다
- 글감 선정 시 "하나의 왜?"로 꿰뚫을 수 있는 회사를 우선한다. 여러 독립적 이야기가 있는 회사는 90점이 구조적으로 어렵다

**작가의 역할**: 관통선을 **뼈대**로 세우고, 모든 소재(숫자, 인물, 사건, 경쟁사)를 이 뼈대에 **살로 붙인다**. 뼈대 없이 살만 붙이면 산만한 글이 된다.

**검증 방법**: 모든 막의 소제목을 나열한 뒤 "이 막을 빼면 관통선이 끊기는가?" 질문. 빼도 관통선이 안 끊기면 → 그 막은 다른 막에 흡수하거나 삭제.

**사례**:
- ✅ SK하이닉스: "5번 죽을 뻔한 회사가 왜 매번 더 강해졌는가" → 모든 막이 이 인과
- ✅ 대한조선: "조선소인데 OPM 24%가 어떻게 가능한가" → 선종 집중→공정 혁신→턴어라운드
- ❌ 현대글로비스: "세 겹의 재무제표" → 물류/해운/지배구조가 병렬 → 85점 천장

### Phase 2: 집필 (재무분석가 → 스토리작가 순차)
- 매 막을 재무분석가(데이터+해석) → 스토리작가(장면+리듬) 순서로 작성
- 매 막 "왜?"로 시작, **관통선 질문의 답을 향해** 흐르기
- 막 수 제한 없음 (5막이든 9막이든)
- **매 막 끝에 다음 막으로의 인과 다리 1문장 필수**

### Phase 3: SVG + 이미지 + 썸네일

**SVG 규칙:**
- 5~7개 (다크 테마 #0a0e1a)
- font-family: `-apple-system, 'Segoe UI', sans-serif`
- 타이틀은 **한국어** 필수 (영문 금지). 형식: "경동나비엔 마진 폭포 — 매출 100% → 영업이익 9.5%"
- 텍스트 fill=#f1f5f9, 부제 fill=#64748b, 강조 fill=#fbbf24
- A등급 #22c55e, C/D등급 #f59e0b/#ef4444

**이미지 생성 (Replicate FLUX):**
- 2~3개. 각 막 도입부에 배치
- API: `REPLICATE_API_TOKEN` (.env에 있음)
- 모델: `black-forest-labs/flux-1.1-pro`
- 호출 방법:
  ```python
  import os, requests, time
  from dotenv import load_dotenv; load_dotenv()
  token = os.getenv('REPLICATE_API_TOKEN')
  headers = {'Authorization': f'Token {token}', 'Content-Type': 'application/json'}
  resp = requests.post('https://api.replicate.com/v1/predictions', headers=headers, json={
      'version': 'black-forest-labs/flux-1.1-pro',
      'input': {'prompt': '...', 'aspect_ratio': '16:9', 'output_format': 'webp', 'output_quality': 90, 'safety_tolerance': 2}
  })
  pred_id = resp.json()['id']
  # poll until succeeded, then download output URL → .webp
  ```
- **Rate limit**: 크레딧 $5 미만 시 분당 6건 제한. 순차 생성 + 간격 10초
- 저장: `assets/{nn}-{slug}.webp` (포스트 번호 prefix)
- 프롬프트 규칙: 영문, "no text" 포함, 제품/장면 사실적 묘사, 브랜드 로고 금지

**썸네일:**
- 1개 (og:image 1200x630, 아바타+제목+배경)
- 저장: `/thumbnails/{stockCode}-{slug}.webp`

**Phase 3 실행 순서 (필수):**
1. SVG 5~7개 먼저 Write tool로 생성 (코드 실행 불필요)
2. FLUX 이미지 2~3개 순차 생성 (rate limit 간격 12초)
3. 본문에 SVG 참조(`![](./assets/{nn}-xxx.svg)`)와 이미지 참조(`![](./assets/{nn}-xxx.webp)`) 삽입
4. SEO audit 실행 → SVG/IMG 카운트 확인
5. **이미지 0개로 나오면 안 됨** — 반드시 FLUX 생성까지 완료 후 커밋

**FLUX 생성 절대 빠뜨리지 않는다.** SVG만 만들고 이미지를 안 만드는 실수 반복 금지.

### Phase 4: 독자 루프 (90점까지 반복)

```
초안 완성
  → 독자 에이전트 평가
  → 90점 미만:
      → 독자가 지적한 약한 막을 3에이전트 병렬 보강
      → 보강 반영 → 독자 재평가
      → 90점 이상이면 발행
  → 90점 이상: 바로 발행
```

**독자 에이전트 평가 항목:**
1. 재밌나? YES/NO + 이유
2. 어디서 집중 끊겼나? 막 번호
3. 관통선 살아있나? 잊히는 막은?
4. "어?" 몇 번? 각각 어디서?
5. 기억에 남는 문장 하나
6. 점수 (100점 만점)

**규칙:**
- 독자에게 글 전체를 넘긴다 (요약 아님)
- "전체적으로 보강"은 금지 — 포인트를 찍어야 결과가 나옴
- 최대 2회 루프. 2회 돌려도 안 되면 구조 재설계
- 독자는 글을 만든 에이전트와 독립

## 5. 글의 구조 규칙

- 막 수 제한 없음
- 매 막 "왜?"로 시작
- **관통선 1개** — 글 전체를 꿰는 질문. 중간에 잊히면 안 됨
- dartlab 코드 자연스럽게 등장 ("직접 해보세요")
- 검증표 필수 (본문 모든 수치의 출처)
- 부채비율 과잉 강조 금지

### 12편 독자 평가에서 발견한 추가 규칙 (2026-04-10)

**표-맥락 규칙**
- 표 앞에 반드시 "왜 이 숫자가 이상한가" 1줄 선행. 숫자 먼저 → 설명 나중 금지
- 부록 재무제표 표는 본문과 단위 통일 (억원이면 억원). 조원/억원 혼용 금지
- 표가 2개 이상 연속이면 표 사이에 해석 문장 1줄 필수 (표만 쌓이면 사보가 된다)

**막 구조 규칙**
- 중복 막 금지 — 같은 데이터를 "BS 관점"과 "CF 관점"으로 2막에 걸쳐 쓰지 않음. 하나의 막에서 교차 관점으로 처리
- 마지막 막이 체크리스트/나열이 되면 안 됨 — 반드시 판단 문장("이것이 다음 재무제표를 바꿀 변수다")으로 마무리
- "5막 (계속)" 같은 분할 금지 — 길면 막을 나누되 고유 번호 부여
- 막 전환 시 인과 연결 1문장 필수 — "3막의 결과가 4막을 가능하게 한 것이다" 수준

**H3 소제목 필수 규칙 (2026-04-12 한전 #25에서 발견)**
- 각 막(H1 `# 제N막`) 안에 **H3 소제목(`### ...`) 3~5개** 필수
- 글 전체 H3 합계 15개 이상 목표
- 소제목이 있으면: 스캔 가독성↑, 내용 밀도 체감↑, SEO H태그 구조↑
- 소제목이 없으면: 막 안이 긴 문단으로 밋밋, 독자가 중간에 위치를 잃음
- 좋은 예: LG이노텍(#18) H3=23, 한온시스템(#17) H3=18, HDC현산(#20) H3=26
- 나쁜 예: SK하이닉스(#01) H3=3, 삼양식품(#02) H3=3, 한전(#25) H3=3
- 소제목 작명: "### 매출원가율 141.6% — 팔수록 적자" (숫자+한 줄 해석) 패턴 권장

**용어 쉽게 풀어쓰기 규칙 (2026-04-13 추가 — 강제)**

독자는 재무제표를 처음 보는 사람이다. 전문 용어를 쓰면 반드시 **같은 문장 안에서** 풀어쓴다.

| 쓰지 말 것 | 이렇게 바꿔라 |
|---|---|
| OPM 6.2% | 영업이익률 6.2% (매출 대비 영업이익 비율) |
| GP 마진 44% | 매출총이익률 44% (물건 팔고 원가 빼면 44% 남는다) |
| OCF | 영업활동현금흐름 (실제 장사해서 들어온 현금) |
| FCF | 잉여현금흐름 (영업현금에서 투자비를 뺀 진짜 남는 돈) |
| CCC -55일 | 현금전환주기 -55일 (물건 팔기 전에 돈이 먼저 들어오는 구조) |
| ICR 0.92x | 이자보상배율 0.92배 (번 돈으로 이자도 못 갚는 상태) |
| ROIC / ROE | 투하자본수익률 / 자기자본수익률 (투자한 돈 대비 이익 비율) |
| DTC | 직접판매 (자사 온라인몰+직영매장) |
| PER / PBR | 주가수익비율 / 주가순자산비율 (주가가 이익/자산 대비 몇 배인가) |
| 지분법손익 | 자회사 이익 중 지분만큼 가져오는 금액 |
| 이연법인세자산(DTA) | 미래에 세금을 덜 낼 수 있는 권리 (회계상 자산) |
| 운전자본 | 영업에 묶여있는 돈 (재고+매출채권-매입채무) |
| 감가상각 | 과거에 산 설비 값을 매년 조금씩 비용으로 깎는 것 (현금은 안 나감) |
| 물적분할 | 자회사를 떼어내 별도 상장시키는 것 |
| 액면분할 | 주식 1장을 N장으로 쪼개는 것 (총 가치는 동일) |

- **괄호 풀이 한 줄 필수**: "OPM 6.2%" 금지 → "영업이익률(OPM) 6.2%" 또는 "매출 대비 영업이익 비율 6.2%"
- **첫 등장 시 풀이, 이후 약어 OK**: 같은 글에서 두 번째부터는 약어만 써도 됨
- **비유 권장**: "감가상각은 20년 전에 지은 공장 값을 매년 장부에서 깎는 것 — 현금은 이미 나갔다"
- **독자 기준**: "내 엄마가 읽어도 이해할 수 있는가?" — NO면 풀어쓴다

**데이터 신뢰 규칙**
- "audit 예정", "추정", "본 글의 범위 밖"으로 빈칸을 남기지 않음. 모르면 "~에 따른 일회성"이라도 1줄 근거를 쓴다
- DCF/적정가를 제시하면 핵심 가정(WACC, 성장률) 1줄 필수
- CF/BS 표의 "—" 행은 실측값으로 채우거나 행 자체를 제거

## 블로그 = 엔진 audit (blog_as_audit)

scan, quant, analysis, search 등 dartlab 엔진을 다루는 블로그 글은 단순 소개가 아니라 **엔진 audit**을 겸한다.

**Why**: 사용자가 명시함 — "이걸하면서 우리 엔진들에 데이터 이상 계산방식 버그를 찾는 일석이조 방식이 될수있다". 글 한 편이 곧 회귀 시나리오. 외부 지식으로 만든 스토리(예: 퀀트 투자로 종목 발견)를 엔진에 부딪혀보면 사용자 기대와 현재 구현의 차이가 드러난다.

**How to apply**:
1. **본문 수치는 100% 실측**. 추정·기억·일반 지식으로 숫자 쓰지 말 것. 한 줄이라도 dartlab 호출 결과 그대로 인용.
2. **호출 로그를 남긴다**. 글 작성 중 발견한 이상치/None/계산 오류는 즉시 별도 메모(`scan_audit.md`, `quant_audit.md` 등)에 기록 → 별도 fix PR.
3. **버그 발견 시 작업 분기**. 단순 표현 문제는 글에서 우회, 진짜 버그는 멈추고 사용자 보고 → 픽스 후 글 재개.
4. **글에 "검증된 호출"을 노출**. 독자가 그대로 복붙해서 같은 결과를 재현할 수 있어야 한다. "이렇게 하면 됩니다" 금지, "이 코드를 실행하세요" 형태.
5. **외부 지식 = 스토리, dartlab = 검증**. 학술 프레임(파마-프렌치, 그레이엄 점수, 모멘텀 등)은 외부에서 가져오고, 한국 시장 적용은 반드시 scan/quant 호출로 확인.
6. **글 작성 = audit 사이클**: 외부 스토리 정리 → 엔진 호출 → 결과 검증 → 이상 발견 시 메모리/이슈 등록 → 글에 반영 또는 우회 → 픽스 별도 진행.

대상 엔진: scan, quant, analysis, credit, macro, search, gather. 단순 Company facade 글(예: 종목코드 하나)도 같은 원칙.

## 기업분석보고서 시리즈 운영 (company_report_series)

## 원칙

기업분석보고서(`blog/05-company-reports/`)는 단일 시리즈로 계속 간다.
서브시리즈/심화 폴더 같은 거 만들지 않는다.

**Why:** 미리 "이 회사 5편 쓰자" 식으로 박아두면 의무감 글이 나옴. 진짜 재밌는 글감이 나올 때만 쓰는 게 콘텐츠 품질에 좋다.

**How to apply:**
- 같은 회사로 여러 편 OK — 종합편, 특정 각도(마진 해부/자본배분 등), 비교편 자유롭게
- 글번호만 04, 05, 06... 계속 늘어남. 카테고리/시리즈 분리 X
- 글감은 미리 로드맵 박지 말고 분석/영상/audit 작업 중 자연스럽게 나올 때 잡음
- 같은 회사 묶기는 frontmatter `stockCode` 필드로 자동 가능 — 새 키 만들 필요 없음
- 글감 출처 예: 영상 만들다 한 컷에 못 넣겠는 깊이, 다른 회사 분석 중 발견한 비교 포인트, 사용자 질문, audit 패턴
