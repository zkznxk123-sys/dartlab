## 0. 한 줄 정의

**카드 = 발명이 아니라 증류.** 사람이 이미 검증한 기업 리포트의 가장 강한 사실을, *약한 훅은 구조적으로 차단하는* Hook Engine으로 골라, 한 화면 0.5초 안에 멈추게 만든다.

dartlab의 비대칭: 일반 크리에이터의 강한 훅은 *신뢰를 깎아* 만들지만, dartlab의 강한 훅은 *데이터가 떠받쳐* 신뢰를 안 깎는다. "이 숫자 진짜야?" → "전자공시 봐" 가 끝나는 순간 충격과 신뢰가 **동시에** 올라간다.

## 1. 카드 시스템 (as-built)

거처: `sns/remotion-sns/src/cards/HookCard.tsx` (sns/ 전체 gitignored). 1080×1350 4:5.

| 레이아웃 | 역할 | 구성 |
|---|---|---|
| `editorial` | 커버(훅) | 모노 사진 + 작은 임팩트 날짜 + 빅 헤드라인(`[[ ]]` 한 구절 강조) + 3~4줄 payoff |
| `editorialBeat` | 내용 비트(키포인트) | 임팩트 키커 라벨 + 헤드라인 + 본문 |
| `editorialStat` | 숫자 모먼트 | 키커 + 거대 숫자(임팩트 컬러) + 한 줄 맥락 |

공통: 좌상단 회사/주제 태그(주식=실명 노출 OK, 자체소식=dartlab 주제명) · 우상단 페이지 점 · 좌하단 dartlab 마크.

**레인별 파이프라인 계약 (강행):**

기술이야기, 기업이야기, 경제이야기, 퀀트이야기는 특히 같은 카드 틀을 공유하지 않는다. 네 레인은 독자 질문, 근거 선택, 숫자 사용법, 마지막 보상이 모두 다르다. `story_brief.json` 에는 현재 레인의 파이프라인 객체 하나만 둔다. 예를 들어 `기업이야기`는 `companyPipeline` 만 쓰고, `economyPipeline`·`quantPipeline`·`technologyPipeline` 을 함께 넣지 않는다. 다른 레인의 객체가 함께 있으면 기획이 섞인 것으로 보고 실패 처리한다.

| 레인 | 첫 장 훅 | 중간 전개 | 마지막 보상 |
|---|---|---|---|
| `기술이야기` | 기능명이 아니라 `누가 어떤 불편을 겪는가`에서 시작한다. 독자가 기능을 모른다는 전제를 둔다. | 기능의 사용 장면, 실제로 할 수 있는 일, 못 하는 일을 순서대로 말한다. 구현 숫자는 기능을 설명한 뒤 증거로만 쓴다. | `무엇을 할 수 있나`, `언제 쓰나`, `어디까지 가능한가`로 닫는다. 투자 지표·다음 실적 관전으로 닫지 않는다. |
| `기업이야기` | 그 회사만의 압박 질문을 건다. 예: 더 벌 수 있나, 왜 같이 뜨나, 비싸도 말이 되나. | 돈이 나는 구조, 숫자가 바뀐 이유, 흔들릴 조건을 연결한다. | 다음 실적·수주·마진·가격·점유율 중 무엇을 봐야 하는지 남긴다. |
| `경제이야기` | 거시 단어가 아니라 내 지갑·기업 숫자·산업 비용에 닿는 질문으로 시작한다. | 큰 흐름이 어떤 경로로 회사 숫자나 생활 가격에 닿는지 보여준다. | 확인할 공개 지표와 영향을 받는 주체를 남긴다. |
| `퀀트이야기` | 숫자가 맞아 보이는 순간 생기는 함정을 건다. | 가정, 표본, 비용, 처음 안 쓴 기간, 반례를 순서대로 풀어쓴다. | 그 숫자를 믿기 전 확인할 조건과 실패 신호를 남긴다. |
| `공시이야기` | 공시 한 줄이 실제로 무엇을 바꾸는지 묻는다. | 공시 원문 문구, 날짜, 조건, 이해관계자를 연결한다. | 원문에서 다시 확인할 문구·날짜·조건을 남긴다. |
| `재무이야기` | 계정명이 아니라 사업에서 새는 돈·쌓이는 돈을 묻는다. | 계정 위치, 현금흐름, 일회성/반복성을 연결한다. | 다음 재무제표에서 볼 계정·비율·현금흐름을 남긴다. |

레인별 색은 전체 배경색이 아니라 강조 구절, 큰 숫자, ref badge, 얇은 선에만 쓴다. 모든 레인의 기본 브랜드 임팩트 컬러는 여전히 dartlab 핑크-레드 `#fb3f6c` 이고, 레인 색은 독자가 게시물 성격을 빠르게 구분하는 신호다.

**시각 규칙 (강행):**
- 사진 = 컬러 webp를 렌더 시 `filter: grayscale(1) contrast(1.06) brightness(0.9)` (사전가공 금지·SSOT 1벌).
- 강조색 = dartlab SNS 임팩트 컬러 **핑크-레드 `#fb3f6c`** (`colors.accentImpact`). **레퍼런스 주황 클론 금지** — 우리 색.
- 강조 = 헤드라인당 정확히 1구절(`[[ ]]`). `accentColor` 만 넣고 `[[ ]]` 를 빼면 강조색은 렌더되지 않는다. 2개+ = 시선 분산 = 강조 0과 동일.
- 첫 줄 ≤ 12~14자(줄넘김 orphan 금지). 헤드라인 90px·본문 34px(2026-06-18 "글자 키움" 반영).
- 가짜 도메인(dartlab.io) 워터마크 금지. 추상 그라데·금융 클립아트 금지(회사 상징 실물만).
- image_gen·Flux 신규 이미지는 반드시 `sns/assets/{sourceCompany}/{semantic}.webp` 공유자산으로 저장하고, 포스트 폴더에는 직접 저장하지 않는다. 캐러셀은 `meta.json.sourceCompany` → `sync_assets.py` → `public/{postId}/` 복사 경로만 소비한다.
- image_gen 프롬프트 끝에는 `Asset key: {sourceCompany}/{semantic}.webp` 를 적는다. 이 키가 빠지면 세션 로그에서 결과를 추출할 수 없어 공유자산 파이프라인이 끊긴다.
- 이미지 수는 고정하지 않는다. `3장 세트` 같은 템플릿 금지. 카드 흐름에서 필요한 의미 장면 수만 만든다. 같은 의미의 배경은 재사용하고, 새 장면은 `사업 정체성`, `작동 방식`, `리스크/관전 포인트`처럼 독자가 다르게 읽는 정보가 있을 때만 추가한다.
- 엔터·미디어 종목 이미지는 회사 로고뿐 아니라 아티스트 얼굴, 특정 그룹을 연상시키는 의상·헤어, 앨범 커버, 콘서트 포스터, 공식 응원봉 형태, 읽을 수 있는 텍스트까지 금지한다. 무대·스튜디오·백스테이지 운영처럼 비식별 사업 장면으로 풀어야 한다.
- 카드 수는 템플릿으로 고정하지 않는다. 인스타 캐러셀은 최대 20장까지 염두에 두되, 기본은 6~12장이고 10장 이상은 새로운 판단 근거가 계속 있을 때만 쓴다.
- 숫자는 뒷부분의 판단 근거로 쓴다. 모든 카드에 숫자를 억지로 넣지 말고, 주가 배수·이익률·목표가 분산·점유율·전력 규모처럼 질문을 바꾸는 숫자만 핵심 카드로 올린다.
- 수치 라벨은 반드시 항목명으로 쓴다: `매출`, `영업이익률`, `수주잔고`, `시가총액`, `주가`, `점유율`. `현재 크기`, `체급`, `무게`처럼 매출·시총·잔고가 섞여 보이는 해석형 라벨은 금지한다.
- 단순 규모 숫자(`매출 4,548억`, `시가총액 1조`처럼 "큰 회사다"만 말하는 숫자)는 `editorialStat`으로 키우지 않는다. 규모는 `editorialBeat` 본문에 넣고, `editorialStat`은 원가율·마진·배수·수주잔고 전환처럼 독자의 질문을 바꾸는 숫자에만 쓴다.
- 첫 번째 장은 정보 요약이 아니라 조회를 결정하는 훅이다. 회사·주제의 가장 강한 질문을 첫 화면에서 찔러야 한다. 하이닉스처럼 `268만원짜리 하이닉스, 더 갈까` 가 먹힌 이유는 가격 자체보다 독자가 이미 묻던 `더 갈 수 있나` 를 정면으로 건드렸기 때문이다. 다른 회사도 같은 문법을 복붙하지 말고, 그 회사의 압박점에 맞춰 `돈이 될까`, `왜 같이 뜰까`, `바로 이익인가`, `무엇이 남나` 같은 고유 질문으로 세게 건다.
- 첫 장 금지형: `알아봅니다`, `살펴봅니다`, `정리합니다`, `소개합니다`, `기본부터`, `무엇인가요`, `세 가지` 로 시작하는 요약형 커버. 이런 문장은 본문 도입으로도 약하고 커버로는 실패다.
- 첫 장 통과형: `audienceQuestion`, `coreAnswer`, `tension`, `soWhat` 중 최소 두 축이 첫 장 헤드라인·서브에 살아 있어야 한다. 첫 장만 보고도 왜 넘겨야 하는지 보여야 한다.
- 마지막 카드는 항상 명확한 마무리다. 결론만 쓰지 말고 레인별 마지막 보상으로 닫는다. `기업이야기`는 다음 지표, `경제이야기`는 공개 지표와 영향 경로, `퀀트이야기`는 믿기 전 확인할 조건, `기술이야기`는 사용 장면·가능한 일·한계를 남긴다. 막연한 여운, "지켜봐야 한다" 단독 문장, 무조건적 매수·매도처럼 읽히는 결론은 실패다.
- 카드 흐름은 맥락 유지가 핵심이다. 앞 카드가 던진 질문을 다음 카드가 받아야 하며, `훅 질문 -> 문제/마찰 -> 근거 표면 -> 작동 방식 -> 현재 상태 -> 어디를 볼지` 순서를 벗어나지 않는다. 서로 연결되지 않는 기능 나열은 실패다.
- 서브 본문은 의도적으로 줄바꿈한다. 독자가 한 호흡에 읽어야 하는 의미 단위만 한 줄에 두고, 자동 줄바꿈으로 단어가 중간에서 끊기게 두지 않는다.
- 기업·주식 에디토리얼 캡션은 스레드에도 올릴 수 있게 짧게 쓴다. 기본 톤은 존댓말이며, `입니다. 아닙니다.` 구조는 필요할 때만 쓴다. 모든 편이 같은 첫 문장 템플릿으로 시작하면 재미가 죽으므로 독자 질문, 마찰, 반전에서 자연스럽게 연다.
- 2026-06-19 기준 릴스는 보류다. 기본 운영은 캐러셀만 만든다. `sns/reels/from_carousel.py {post}` 는 운영자가 명시적으로 릴스를 다시 열 때만 쓴다. 보류 중에도 원본 이미지는 공유자산 1벌만 유지한다.
- 릴스를 다시 열 경우, 상단 150px은 인스타 그리드/오버레이 안전영역으로 비워 둔다. dartlab 마크는 캐러셀과 같은 `아바타 + dartlab` 구조를 좌하단에 두고, 본문은 하단 35~40% 영역에 자연스럽게 이어지게 한다. 딱딱한 컷 전환만 이어 붙인 영상은 실패다.

**캐러셀 운영 루프 (강행):**
1. 한 제작 사이클은 `기획 3인 + 마케팅 3인 -> 작가 3인 -> 평가 5인 + 마케팅 3인 + 아무것도 모르는 독자 5인 -> 피드백 반영 -> 평가 5인 + 마케팅 3인 + 아무것도 모르는 독자 5인 재평가` 순서로만 통과한다.
2. 주제는 매번 전문 에디터 토론으로 고른다: `News Planner`, `Context Planner`, `Evidence Planner`. 여기에 마케팅 전문가 3인 `Product Marketing Lead`, `Growth Hook Lead`, `Trust Claims Lead` 를 반드시 붙인다.
3. 초안은 작가 3인이 나눠 쓴다: `Hook Writer`는 첫 장 훅, `Structure Writer`는 카드 흐름, `Caption Writer`는 캡션·스레드 마무리를 책임진다. 작가 라운드가 없으면 카드가 있어도 발행 후보가 아니다.
4. 평가는 최소 5역을 붙인다: `Context Editor`, `Skeptical Reader`, `Caption Editor`, `Fact Check`, `Visual QA`. 여기에 마케팅 3인과 초보 독자 5인이 평가 라운드와 재평가 라운드 모두 참여한다.
5. 한 사이클은 6개 레인의 균형을 본다. 기본 묶음은 `기술이야기 -> 기업이야기 -> 경제이야기 -> 공시이야기 -> 퀀트이야기 -> 재무이야기` 순서다. 확장 백로그는 `expanded-carousel-cycle-20.md`가 SSOT다.
6. 후보는 `첫 장의 강한 훅 문장` · `실제 사실` · `레인별 마지막 보상` 이 동시에 있을 때만 통과한다. 첫 장 훅은 주제별로 새로 기획한다. 조회가 잘 나온 과거 문법을 그대로 복제하지 않는다.
7. 최근성 있는 기업·경제·정책·공시 내용은 발행 전 반드시 브라우징 또는 1차 자료로 재확인한다. 확인 전 후보는 `live-check 필요` 상태다. DartLab 계산/기능 내용은 repo 파일, 테스트, 진행 문서 중 최소 2개 이상으로 확인한다.
8. 큰 숫자는 숫자가 질문을 바꿀 때만 `editorialStat` 로 쓴다. 단순 회사 규모 숫자는 `editorialBeat` 안에 넣는다. 숫자 단위가 따로 떠 보이면 실패다.
9. 본문 줄바꿈은 사람이 의도적으로 넣는다. 렌더러는 한국어 단어를 음절 단위로 찢지 않도록 `keep-all` 계열 줄바꿈을 쓴다. 자동 줄바꿈에 맡겨 단어가 쪼개지거나, 의미 단위가 화면 중간에서 끊기면 실패다.
10. 메인 헤드라인은 크게 간다. 짧은 훅은 108~116px, 중간 훅은 100~108px, 긴 줄은 96px까지 낮추되 줄바꿈을 먼저 고쳐서 키운다. `아니/라`, `AI보/다`처럼 단어가 쪼개지는 크기 확대는 실패다.
11. 본문은 36~40px 기본이다. 4줄 이상 필요한 설명은 카드를 나누거나 문장을 줄인다. 본문을 작게 만들어 화면에 우겨넣는 방식은 실패다.
12. 렌더 후 모든 카드를 이미지로 직접 열어 본다. 배치 렌더, 연락시트, 파일 생성 성공만으로는 통과가 아니다. `01-hook.png`부터 마지막 카드까지 한 장씩 확인하고, 고아 줄바꿈, 단어 중간 끊김, 작은 본문, 모호한 수치 라벨, 어색한 단위, 브랜드 누락이 있으면 다시 쓴다.

**맥락·캡션 게이트 (강행):**
1. `hook.json` 작성 전 `story_brief.json` 을 먼저 쓴다. 필수 필드: `lane`, `audienceQuestion`, `coreAnswer`, `tension`, `soWhat`, `whereToLook[]`, `captionThesis`, `captionClose`, `forbiddenAngles[]`, `imageCountDecision`. 레인별 추가 필드는 `technologyPipeline`, `companyPipeline`, `economyPipeline`, `disclosurePipeline`, `quantPipeline`, `financialPipeline` 처럼 별도 객체로 둔다. 현재 레인 객체 하나만 채우고 다른 레인 객체는 넣지 않는다.
2. `audienceQuestion` 은 독자가 실제로 묻는 말이어야 한다. 예: `그래서 이 회사는 더 벌 수 있나`, `이 기능이 실제로 뭐를 막나`. 내부 기능명이나 회사 소개 문장은 질문이 아니다.
3. `coreAnswer` 는 한 줄 결론이다. 카드 전체가 이 답을 향해 가야 한다. 중간 카드가 흥미롭더라도 결론으로 돌아오지 않으면 삭제하거나 순서를 바꾼다.
4. `soWhat` 은 "그래서 독자가 무엇을 다르게 봐야 하나" 를 쓴다. `재밌다`, `주목된다`, `지켜봐야 한다` 단독 문장은 실패다.
5. `whereToLook[]` 은 마지막 카드와 캡션 마지막 문단에 모두 살아 있어야 한다. 단, 의미는 레인별로 다르다. `기술이야기`에서는 사용 장면·가능한 일·한계이고, `기업이야기`에서는 다음 숫자·흔들릴 조건이며, `경제이야기`에서는 공개 지표·영향 경로이고, `퀀트이야기`에서는 가정·표본·비용·반례다. 표현은 줄여도 되지만, 마지막 보상이 바뀌면 실패다.
6. `caption.txt` 는 카드 요약문이 아니다. 3문단 안에서 `질문/마찰 -> 핵심 답 -> 어디를 볼지` 로 닫는다. 기본 길이는 250~450자, 긴 스레드는 `threads.txt` 로 분리한다.
7. `threads.txt` 는 스레드에 바로 올릴 수 있는 독립 글이다. 캡션보다 길 필요는 없지만, 질문·핵심 답·확인 포인트가 스레드만 읽어도 남아야 한다. 새로운 결론을 만들지 않는다. 캡션과 다른 관전 포인트를 말하면 실패다.
8. `comment_pinned.txt` 는 근거와 주의문만 둔다. 카드의 결론을 다시 설득하거나 투자 판단처럼 보이는 문장은 쓰지 않는다.
9. 공개 문구는 독자가 바로 이해하는 말로 쓴다. 내부 약어와 전문어는 먼저 풀어 쓴다. 예: `OOS`는 `처음에 안 쓴 기간`, `B&H`는 `그냥 들고 갔을 때`, `slippage`는 `원하는 가격에 바로 샀다는 가정`, `CAPEX`는 `설비투자`, `FCF/OCF`는 `현금이 남는지/본업 현금`, `Macro Lens`는 `경제 뉴스가 회사 숫자에 닿는 길`로 쓴다.
10. 공개 표기는 회사명·주제명·종목코드 6자리로 완성한다. `D01`, `d01`, `E22`, `X03` 같은 내부 분류번호는 폴더명, 제작 메타데이터, QA 기록에서만 관리한다. 발행 순서는 번호순이 아니므로 렌더러와 작성자는 폴더명 앞부분을 상단 배지, 카드 본문, 릴스 자막, 캡션, 스레드, 고정 댓글의 공개 문구로 가져오지 않는다.

**발행 전 리뷰 에이전트 (강행):**
1. 기획 3인은 `News Planner`, `Context Planner`, `Evidence Planner` 로 고정한다. 이들은 주제, 독자 출발점, 근거 숫자를 정한다.
2. 작가 3인은 `Hook Writer`, `Structure Writer`, `Caption Writer` 로 고정한다. 이들은 첫 장, 카드 흐름, 캡션/스레드를 각각 책임지고 `writerRound[]`에 초안·수정·유지할 문장을 남긴다.
3. 최소 평가 5역을 붙인다: `Context Editor`(스토리 흐름), `Skeptical Reader`("그래서 뭐?" 반응), `Caption Editor`(캡션/스레드), `Fact Check`(근거·과장), `Visual QA`(줄바꿈·단위·시선 흐름).
4. 마케팅 전문가 3인은 기획과 평가에 반드시 포함한다: `Product Marketing Lead`(누가 왜 쓰는지), `Growth Hook Lead`(첫 장 후킹), `Trust Claims Lead`(홍보 문구의 과장·근거). `evaluationRound1[]`과 `evaluationRound2[]`에도 pass 근거가 없으면 통과가 아니다.
5. 아무것도 모르는 독자층 5인을 평가 루프에 붙인다: `First-time Reader`, `Non-user Reader`, `Busy Feed Reader`, `Skeptical Buyer`, `Caption-only Reader`. 이들은 기능명·전문어·내부 약어를 모른다는 전제로 `그래서 뭔 말인가`, `그래서 뭘 할 수 있나`, `캡션만 봐도 남는가`를 본다.
6. `editorial_loop.json`의 역할 세트는 정확해야 한다. `planningRound[]`는 기획 3인+마케팅 3인, `writerRound[]`는 작가 3인, `evaluationRound1[]`와 `evaluationRound2[]`는 기본 평가 5인+마케팅 3인+무지 독자 5인을 각각 정확히 한 번씩만 담는다. 누락, 중복, 임의 역할 추가는 실패다.
7. 리뷰 결과는 `editorial_review.md` 에 남긴다. 각 역할은 `Verdict: pass`, `Message:`, `Checks:`, `Blocking issues: none` 형식으로 쓴다. 리뷰 본문은 `oneLineMessage` 와 `whereToLook[]` 를 실제 근거로 다뤄야 한다.
8. 사람이 보기에 "보류 가능"이어도 `checkCarouselEditorial.py` 가 실패하면 발행 상태는 보류다. 전문 에이전트 평가는 기계 게이트를 대체하지 않는다.
9. 전문 에이전트가 OK를 냈다는 표현은 쓰지 않는다. 유효한 판정은 `story_brief.json`, `caption.txt`, `threads.txt`, 마지막 카드, `editorial_review.md`, 렌더 이미지가 같은 결론과 같은 관전 포인트를 공유할 때만 가능하다.
10. `Skeptical Reader` 는 반드시 다음 질문에 답한다: `첫 장만 보고 왜 넘겨야 하나`, `마지막 장까지 보고 무엇을 저장하나`, `캡션만 읽어도 관전 포인트가 남나`.
11. `Caption Editor` 는 캡션이 카드 본문을 복붙한 요약인지 검사한다. 요약이면 실패다. 캡션은 독자가 게시물을 저장하거나 스레드로 옮길 이유를 남겨야 한다.
12. `Visual QA` 는 모든 장을 실제 이미지로 보고 줄바꿈·단위·시선 흐름을 기록한다. 직접 확인 기록은 날짜별 QA 문서에 남긴다.
13. 검사 명령은 렌더 전후 모두 실행한다.

```bash
uv run python -X utf8 sns/scripts/checkCarouselEditorial.py <post-folder>
uv run python -X utf8 sns/scripts/auditCarouselEditorialBatch.py
```

**2026-06-20 품질 감사 결론:** 이전 `passed-visual` 표기는 현재 게이트 통과를 의미하지 않는다. 최근 3일 대상 35개는 강화된 `editorial_loop.json` 기반 게이트, 첫 장 훅 게이트, 릴스 공개언어 게이트를 통과했다. PNG 검수는 배치 결과가 아니라 날짜별 직접 QA 문서에서 모든 카드를 한 장씩 연 기록으로만 인정한다.

## 2. Hook Engine — "확실한 후킹"의 핵심 (2라운드 토론 산물)

후킹은 운빨이 아니다. **generate-many → score → pick → (약하면 스킵)** 의 결정론 구조가 *발행된 모든 카드의 훅 하한선*을 보장한다.

### 2.1 후크 taxonomy 8종

| Type | 심리기제 | 공식 | dartlab 예시 |
|---|---|---|---|
| **A 숫자충격** | 직관 척도 초과 → 불신 → 검증욕구 | `[열등상태]가 [시간압축]에 [절대숫자]` | "망하기 직전 회사가 한 분기에 **19조**" |
| **B 역설** | 양립불가 두 사실 → 부조화 해소욕구 | `[A인데] [정반대 B]` | "순손실인데 영업이익 사상 최대"(고려아연) |
| **C 버려진것의 반전** | 권위자 오판 + 약자 승리 | `[권위자]가 [헐값]에 버린 게 [거대가치]` | "삼성이 8,400억에 버린 무기가 수주 37조"(한화) |
| **D 꼴등의 역전** | 정체성 전복 | `[최하위]가 [거대정체성]이 되기까지` | "라면 꼴등이 글로벌 거인"(삼양) |
| **E 미스터리 질문** | 닫고싶은 열린 루프 | `[주체]는 왜 [의외행동]했을까` | "미래에셋은 왜 한 주도 못 받았을까" |
| **F 의외의 정체** | "안다고 생각한 것" 전복 | `[익숙한 것]은 사실 [전혀 다른 것]` | "주가의 90%는 사이클이 결정"(HMM) |
| **G 권력/불의** | 분노·정의감(최강 공유동력) | `[강자]가 [약자]에게서 부당하게` | 데이터가 정당화할 때만 |
| **H 정체성 위협** | 손실회피 + 자기관련 | `너 같은 [독자]가 먼저 당한다` | 시리즈 톤 한정·품격 주의 |

**주력 = A·B·C·D** (검증 숫자로 자기완결). G·H는 정직성/품격 리스크 최대 — 데이터가 그 감정을 직접 정당화할 때만.

### 2.2 후보 생성 (결정론, 4개 기존 자산 소비 — 신규 계산 0)

| 소스 | 위치 | → 후크 type |
|---|---|---|
| frontmatter `ai.*` | `blog/05-company-reports/*/index.md` (`keyMetrics·strengths·weaknesses·archetype`) | A 숫자충격 · B 역설(strength×weakness 충돌) |
| `detectThreads` | `story/narrative.py` (NarrativeThread{title,evidence,severity}) | D/F 의외정체 · 리스크 |
| `narrate._classify` | `story/narrate.py` (값→임계 레이블) | A 숫자충격 임계 판정 |
| `getRankOrBuild` | `scan/screen/rank.py` (RankInfo) | C/극단치 "업종 1위/상위 10%" |

**fallback 사다리** (밋밋한 대형주 보장): S1 절대숫자충격 → S2 횡단순위 → S3 변화율 → S4 정체반전 → S5 미스터리질문(`buildActTransitions`가 항상 의문형 1개 생성 = 0 안 됨). 5단계 전부 비면 **카드 스킵**(데이터 없는 종목).

### 2.3 reject-gate (점수 이전 hard block)

| 게이트 | 차단 | 성격 |
|---|---|---|
| G1 무숫자·추상 | 숫자 0 + 추상명사("성장/혁신/저력")만 | 강도 |
| G2 첫 줄 길이 | > 18자 | 강도 |
| G3 강조 구절 | 0개 또는 3개+ | 강도 |
| G4 과장어 | "세계급/유일/최고/전무후무" NEVER-CLAIM grep | **정직성·우회불가** |
| G5 막연 서술 | "회사가 ~했다" 수치·고유명사 0 | 강도 |
| G6 갭 자폭 | 첫 줄에 원인+결과 모두 명시 | 강도 |
| G7 근거 미검증 | 인용 수치가 panel/scan ref에 매칭 안 됨 | **정직성·우회불가** |

### 2.4 scoring 루브릭 (0~100)

`숫자충격 22 + 호기심갭 20 + 의외성 18 + 감정전하 14 + 1초가독 12 + 구체성 8 + 강조선명 6`. (A+B+C+D = 74 = 충격·갭·역설·생존에 집중. 약한 훅은 정확히 이 4축이 빈다.)

### 2.5 pick 임계

- **≥ 75** : 자동 발행 (강함 보장)
- **60~74** : 보강 큐 — 사람이 더 강한 대안으로 끌어올리거나 *천재 훅 구제* override
- **< 60** : 회사 스킵 (약한 발행보다 정직한 스킵)

보장의 정체 = "최고 훅 강제"가 아니라 **"약한 훅 발행 거부"** = 훅 하한선.

### 2.6 A/B 학습 루프 (경량·과설계 금지)

발행 후 인스타 **저장율·공유율**(좋아요·노출은 노이즈)을 훅 type별로 수집 → 월 1회 사람이 표 한 장(type×축×저장율) 보고 **루브릭 7축은 고정, 가중치만 ±10% 캡** 보정(`rubric_v` 태그). ML·자동분기 금지. `OutcomeLog` 패턴 재사용. override 발행물 성과 = 루브릭 진화의 입력.
