# 00 — 제품 PRD

상태: PRD v1.0 (6 분야 전문 패널 검증·통합 후). [00-design-brief.md](00-design-brief.md) 원안을 패널이 정정한 확정본.

---

## 1. 문제 정의

DartLab 은 전자공시·시세 데이터를 한 화면에 모은 연구 터미널·리포트·블로그다. 그러나 **모든 소비가 pull**
이다 — 사용자가 와서 봐야 안다. 변화(새 글·카드 발행, 공시·주가·IPO 이벤트)가 생겨도 먼저 알려줄 채널이 없다.

정적 사이트(GitHub Pages)는 런타임 쓰기가 불가하므로, 알림을 *쏘는 수신단*이 없다. 받는 쪽(PWA 웹 푸시)도
**셸 캐시만 있고 푸시 수신 파이프는 미존재**다(아래 §5 정정).

## 2. 사용자

### 2.1 1차 사용자
- 특정 회사·산업·테마를 추적하는 개인 투자자·연구자 — 새 공시·실적·수주·IPO 를 놓치고 싶지 않다.
- DartLab 블로그·카드뉴스 구독자 — 새 발행을 폰으로 받고 싶다.

### 2.2 비대상 (현 단계)
- 계정·로그인 기반 개인화를 원하는 사용자(본 플랫폼은 **계정 0**).
- iOS 16.4 미만 / PWA 미설치 사용자(웹 푸시 물리적 불가 — 설치 유도만).

## 3. 제품 약속

### MUST
- 새 글·카드 발행 시 구독자 기기로 알림이 온다(P1).
- 알림 구독·해지는 **계정 없이** 익명 endpoint 1개로만. 모두 끄면 즉시 삭제.
- 개인 조건(내 종목·임계)은 **서버에 저장하지 않는다** — 로컬 소유.
- `pip install dartlab` 한 사용자는 알림 기능과 무관하게 기존대로 작동(알림은 순수 추가 표면).
- 발송 트리거(`/send`)는 인증된 주체(러너)만 — 누구나 구독자에게 푸시 못 쏜다.

### SHOULD
- 공개 시장 이벤트 토픽(신규수주·IPO 등)을 구독해 브로드캐스트로 받는다(P2).
- 개인 종목 알림 — 기존 워치리스트(`watchlist.svelte.ts`)에 '알림 ON' 비트만 추가(P3, 새 화면 신설 금지).
- 알림 본문은 외부 원문 인용이 아니라 **구조화된 사실 카드**(회사·이벤트유형·시각)로.
- 빈도 cap·조용한 시간(기본 22:00~08:00 묶음)·중복 억제로 알림 피로 방지.

### KILL (하지 않는다)
- D1 에 개인 조건(종목·임계) 컬럼 신설 — 영구 0(재식별 surface).
- 새 L2 "왓처 엔진" 신설 — 도메인 격리·graph 회귀·engine-add 게이트 비용.
- P1/P2 에 왓처 레지스트리 추상 선설치 — N=5·형태 제각각 = YAGNI.
- 별도 만료 endpoint 청소 cron — `/send` 응답 404/410 inline DELETE 로 자가청소.
- 알림 권한 콜드 자동 팝업 — 1회 거부 = 브라우저 영구 차단.

## 4. 핵심 아키텍처 (요약 — 상세 [01](01-architecture.md))

3계층: **왓처(dartlab 라이브러리 지능) · 러너(GHA cron 스케줄) · 허브(Cloudflare thin 배달)**.
감지 지능은 dartlab(gather·scan SSOT 직독, 베이크 0), Cloudflare 는 구독 저장 + 발송만.

왓처의 *물리적 자리는 단계별로 다르다*: P1-P2 는 러너 plain 함수(src 밖, 게이트 비대상), P3 에서
두 번째 소비자(로컬 데몬)가 같은 평가 함수를 재사용함이 실증될 때 `synth/watch` 로 발견적 추출.

## 5. 브리프 원안 정정 (패널 확정)

| 원안 주장 | 사실 | 영향 |
|---|---|---|
| "받는 쪽 service-worker 이미 라이브" | `service-worker.ts` 는 install/activate/fetch만. push/notificationclick/pushsubscriptionchange **0**. landing 전역 pushManager/VAPID/requestPermission **0건** | P1 = 수신 스택 5종 신규 구축 |
| "왓처 = dartlab 라이브러리(자리 미정)" | 새 L2 엔진은 회귀. recipe 도 아님 | P1-P2 러너 함수 → P3 synth/watch |
| "신규수주 = order_flow_scan 장기 합류 후보" | `scan.orders` 축 **이미 본진 졸업**(한글 alias 4종 라이브). 메모리 표기 stale | P2 첫 레퍼런스, 새 데이터 0 |
| "왓처 타입 레지스트리부터" | N=5 형태 제각각 → 통일 환상 | 레지스트리는 P3 발견적 추출 |
| (역할문구) "D1 개인조건 테이블" | endpoint+종목=재식별 | D1 개인조건 컬럼 영구 0 |

## 6. 성공 기준

- P1: 구독자가 새 글·카드 발행 알림을 실제 기기에서 수신(실측 1대 이상, iOS standalone 포함).
- P2: 공개 토픽(신규수주 등) 브로드캐스트가 중복·야간 폭주 없이 동작(dedup·cap 실측).
- P3: 개인 종목 알림이 계정 0·서버 미저장으로 동작(로컬 소유 실증).
- 전 단계: 새 인프라 자원 = pushHub Worker 1개 + D1 1개뿐(러너=기존 GHA 재사용).
