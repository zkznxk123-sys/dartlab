# Experiments

**주체**: 실험 워크플로우 (`experiments/NNN_*/`).
**현재**: 폴더 분리 + STATUS.md · 실험 산출물 gitignored 경로 · 검증 결과는 본체 흡수 또는 archive.
**방향**: 실험→프로덕션 흡수 추적 자동화 · 실험 결과 review audit 연계.

가설 검증 → 결론 → 흡수 판단의 실험 체계.

## 폴더 규칙

- `experiments/XXX_camelCaseName/` — 실험별 하위 폴더 분리
- 각 폴더에 `STATUS.md`로 현황 기록
- 실험 코드는 반드시 실험 폴더 안에서 관리한다

## 파일 규칙

- 네이밍: `XXX_camelCaseFeature.py` (001부터)
- 독립 실행 필수: `if __name__ == '__main__':` 블록
- 결과는 docstring에 직접 기록한다 (별도 결과 파일 불필요)
- 실패한 실험도 "왜 실패했는지" 결론 명시

## 절대 규칙

- **파일을 만들면 반드시 실행하고 결과를 채운다** — 같은 턴에서 완결
- 모든 `.py`는 열었을 때 목적/가설/방법/실제 결과/결론이 완결된 상태여야 한다
- 아직 실행할 수 없으면 파일을 만들지 않는다

## Docstring 구조 (필수)

```
실험 ID: XXX
실험명: 제목

목적:
- 해결하려는 문제, 기대 인사이트

가설:
1. 정량적 기준 포함한 가설

방법:
1. 구체적 절차, 사용 데이터, 비교 대상, 측정 지표

결과 (실험 후 작성):
- 수치 테이블

결론:
- 채택/기각 판단

실험일: YYYY-MM-DD
```

## 흡수 규칙

- **실험 완료 ≠ 흡수 승인** — 사용자가 명시적으로 승인한 후에만 정식 코드에 반영한다
- 검증이 끝났더라도, 실험 폴더 밖 파일은 승인 후에 수정한다
- 매핑 데이터(accountMappings.json 등) 기여는 실험 기반으로만 수용

## 완료 기준

- docstring `결과`와 `결론`이 실제 실행 결과 기준으로 채워져야 완료
- `STATUS.md`에도 같은 세션에서 핵심 수치와 판단 반영
- 사용자가 파일만 열어도 실험의 상태와 판단을 이해 가능
- 결과가 불충분하면 "미완료 실험" 취급 — 흡수 근거로 사용 불가
