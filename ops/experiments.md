# Experiments

**주체**: 실험 워크플로우 (`experiments/NNN_*/`).
**현재**: 폴더 분리 + STATUS.md · 실험 산출물 gitignored 경로 · 검증 결과는 본체 흡수 또는 archive.
**방향**: 실험→프로덕션 흡수 추적 자동화 · 실험 결과 story audit 연계.

가설 검증 → 결론 → 흡수 판단의 실험 체계. 각 섹션은 **"이렇게 한다"** 명제로 열고, 반복된 실수는 섹션 하단 **"반복 실패"** 에 정리한다.

---

## 1. 폴더 — `experiments/XXX_camelCaseName/` + `STATUS.md` 로 정리한다

- `experiments/XXX_camelCaseName/` — 실험별 하위 폴더 분리.
- 각 폴더에 `STATUS.md` 로 현황 기록.
- 실험 코드는 실험 폴더 안에서 관리한다.

---

## 2. 파일 — `XXX_camelCaseFeature.py` 로 독립 실행 가능하게 쓴다

- 네이밍 — `XXX_camelCaseFeature.py` (001 부터).
- 독립 실행 필수 — `if __name__ == '__main__':` 블록.
- 결과는 docstring 에 직접 기록한다 (별도 결과 파일 없음).
- 실패한 실험도 "왜 실패했는지" 결론 명시.

---

## 3. 파일 생성 — 만들었으면 같은 턴에 실행하고 결과를 채운다

- 파일을 만들면 **반드시 실행하고 결과를 채운다** — 같은 턴에서 완결.
- 모든 `.py` 는 열었을 때 목적·가설·방법·실제 결과·결론이 완결된 상태.
- 아직 실행할 수 없으면 파일을 만들지 않는다.

**반복 실패** — 파일만 만들고 실행 미룸. 다음 세션에 열었을 때 결과 비어있어 의미 없음.

---

## 4. Docstring — 이 구조로 채운다

```
실험 ID: XXX
실험명: 제목

목적:
- 해결하려는 문제, 기대 인사이트

가설:
1. 정량적 기준 포함한 가설

방법:
1. 구체적 절차, 사용 데이터, 비교 대상, 측정 지표

결과 (실험 후 작성):
- 수치 테이블

결론:
- 채택/기각 판단

실험일: YYYY-MM-DD
```

---

## 5. 흡수 — 사용자 승인 후에만 본체에 반영한다

- **실험 완료 ≠ 흡수 승인** — 사용자가 명시적으로 승인한 후에만 정식 코드에 반영.
- 검증이 끝났더라도, 실험 폴더 밖 파일은 승인 후에 수정한다.
- 매핑 데이터 (`accountMappings.json` 등) 기여는 실험 기반으로만 수용.

**반복 실패** — 실험이 좋게 나왔다고 자체 판단으로 본체에 바로 머지. 사용자 승인 게이트가 SSOT.

---

## 6. 완료 기준 — docstring + STATUS.md 양쪽 채워야 "완료"

- docstring `결과` 와 `결론` 이 실제 실행 결과 기준으로 채워져야 완료.
- `STATUS.md` 에도 같은 세션에서 핵심 수치와 판단 반영.
- 사용자가 파일만 열어도 실험의 상태와 판단을 이해 가능해야 한다.
- 결과가 불충분하면 "미완료 실험" 취급 — 흡수 근거로 사용하지 않는다.

---

## 요약 — 명제 6 줄

1. 실험은 `experiments/XXX_camelCase/` + `STATUS.md` 폴더 구조로 정리한다.
2. 파일은 `XXX_camelCaseFeature.py`, `if __name__ == '__main__'` 로 독립 실행 가능하게.
3. 파일 만들면 같은 턴에 실행하고 결과·결론을 docstring 에 채운다.
4. Docstring 구조 (ID·실험명·목적·가설·방법·결과·결론·실험일) 를 완결 상태로 유지한다.
5. 실험 완료 ≠ 흡수 승인. 사용자 명시 승인 후에만 본체 반영한다.
6. docstring + STATUS.md 양쪽이 채워져야 완료. 불충분하면 흡수 근거 없음.
